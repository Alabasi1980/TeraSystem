# Tera Pricing Policy

**الإصدار:** v0.1 — Draft — Calibration Required  
**آخر تحديث:** 2026-07-03  
**مصدر الحقيقة:** هذا الملف  
**الجهة المنفذة:** TeraClientEngagementAgent (مسودات فقط)  
**الموافقة النهائية:** Majed

---

## 1. المبادئ الأساسية

1. **العدالة**: نفس الخدمة = نفس السعر عبر مشاريع مختلفة.
2. **الشفافية**: للزبون يُعرض سعر مبسط. المصفوفة الداخلية والأسس الحسابية لا تُعرض.
3. **الانسجام**: جميع الأسعار تصدر عن هذا الملف كمرجع وحيد — لا أسعار عشوائية أو تقديرات غير موثقة.
4. **المرونة المدارة**: v0.1 — يُضبط بعد أول 3 عروض أو 2 مشاريع.
5. **الأمان**: TCEA يصدر Draft فقط. Majed هو من يعتمد السعر النهائي.

---

## 2. نطاق التطبيق (Scope) — v0.1

- هذا الإصدار يغطي: **Custom Software Development فقط**
  - تطبيقات ويب، تطبيقات موبايل، متاجر إلكترونية، أنظمة إدارية.
- **لا يغطي:** Consulting ERP (SAP/D365) — سيُبنى نموذج منفصل لاحقاً.
- أي مشروع ERP في الفترة الانتقالية يُسعّر يدوياً بـ Day Rate أو Milestone-Based.

---

## 3. معادلة التسعير الداخلية

```
Feature Base Price
× Complexity Factor (1.0 / 1.5 / 2.5 / 3.5+)
+ Integration Costs
+ Risk Buffer (0% / 10% / 20%)
+ QA / Documentation Allowance
+ Margin (20%)
= Client Quote Price
```

**Base Price** يُحتسب كالتالي:  
`ساعات التطوير المتوقعة عند مستوى Simple × السعر الداخلي للساعة`

السعر الداخلي للساعة: **15–25 JOD** (تقدير تجريبي — يُضبط بعد أول مشروعين).

## 4. Rubric تصنيف التعقيد (Complexity Scoring Rubric)

كل ميزة/شاشة تُصنّف بجمع نقاط من المعايير التالية:

| المعيار | 0 نقاط | 1 نقطة | 2 نقطتان | 3 نقاط |
|---------|--------|--------|----------|--------|
| عدد جداول البيانات المرتبطة | 0-1 جدول | 2-3 جداول | 4-6 جداول | 7+ جداول |
| منطق الأعمال (Business Logic) | لا يوجد | شرط واحد | 2-3 شروط/فروع | Workflow معقد أو منطق مالي |
| مستويات الصلاحيات | دور واحد | 2-3 أدوار | 4-5 أدوار | 6+ أدوار مع صلاحيات دقيقة |
| تكاملات خارجية (API) | 0 | 1 | 2 | 3+ |
| تقارير / تصوير بيانات | لا يوجد | تقرير بسيط (قائمة) | تقرير مع رسوم بيانية | لوحة قيادة تفاعلية |
| إشعارات / أحداث زمنية | 0 | إشعار بريدي | إشعارات + SMS | إشعارات + SMS + فوري |

| مجموع النقاط | مستوى التعقيد | المعامل |
|-------------|---------------|---------|
| 0-3 | **Simple** | 1.0x |
| 4-7 | **Medium** | 1.5x |
| 8-12 | **Complex** | 2.5x |
| 13+ | **Critical** | 3.5x أو تسعير خاص |

(هذا الـ Rubric تجريبي — يُبسّط أو يُعقّد حسب الحاجة بعد أول 5 مشاريع.)

## 5. مصفوفة التسعير الأساسية

- لكل نوع تطبيق: **Base Price** (يُحتسب من الساعات المتوقعة × السعر الداخلي).
- للوحدات الإضافية: **Add-on Prices**.
- معاملات التعقيد تطبق على كل ميزة أو شاشة.
- **المصفوفة الافتراضية أدناه أمثلة توضيحية تُضبط بعد أول مشاريع — وهي داخلية ولا تُعرض للزبون.**

### أمثلة توضيحية (داخلية — قابلة للمعايرة)

| الميزة / الوحدة | Base Price (Simple) | Medium (×1.5) | Complex (×2.5) |
|----------------|--------------------:|:------------:|:--------------:|
| Core Web App (أساسي: شاشات CRUD + تسجيل دخول) | ~1,000 JOD | ~1,500 JOD | ~2,500 JOD |
| وحدة إدارة مخزون | ~700 JOD | ~1,050 JOD | ~1,750 JOD |
| تقارير ولوحة بيانات | ~350 JOD | ~525 JOD | ~875 JOD |
| إدارة الصلاحيات والأدوار | ~250 JOD | ~375 JOD | ~625 JOD |
| تكامل API خارجي (واحد) | ~300 JOD | — | — |
| بوابة دفع إلكتروني | ~400 JOD | ~600 JOD | ~1,000 JOD |

| النوع | الحد الأدنى |
|-------|------------:|
| Web app مخصص | 500–700 JOD |
| Business system | 1,200 JOD |

(قابلة للمعايرة بعد أول 3 مشاريع.)

## 7. Discovery / Paid Analysis

- للمشاريع غير الواضحة أو عالية المخاطر: **Analysis Package** اختياري.
- **السعر المرجعي:** 50–100 JOD أو 5% من قيمة المشروع التقديرية (أيهما أقل).
- يُخصم من قيمة المشروع إذا تم التعاقد خلال 30 يوماً.
- يُحتسب Discovery ضمن Risk Buffer: إذا كان `Critical/Unknown`، الـ Discovery إلزامي قبل التسعير.

---

## 8. Risk Buffer — معايير التصنيف

| المعيار | Low (0%) | Medium (+10%) | High (+20%) |
|---------|----------|---------------|-------------|
| وضوح المتطلبات | واضحة ومفصّلة | عامة لكن مفهومة | غامضة أو متغيرة |
| تقنية التطوير | مألوفة ومجربة | جديدة نسبياً | غير مألوفة أو beta |
| تاريخ الزبون | عميل سابق أو موصى به | عميل جديد بمعلومات واضحة | عميل جديد غير معروف |
| اعتماد على API خارجي | 0 | 1–2 معروفة | 3+ أو API غير موثقة |
| الجدول الزمني | 8+ أسابيع | 4–8 أسابيع | أقل من 4 أسابيع |

**قاعدة التصنيف:**
- 3 معايير High أو أكثر → **High Risk** (+20%)
- 2 معيار High → **Medium Risk** (+10%)
- غير ذلك → **Low Risk** (0%)
- إذا كان `Critical/Unknown` → يتطلب Discovery أولاً

---

## 9. هامش الربح — ملاحظة للمعايرة

- الـ **20%** الحالية: تقدير أولي.
- بعد أول مشروعين فعليين: يُعاد حساب الهامش ليشمل:
  - التكلفة المباشرة (وقت البرمجة)
  - التكلفة غير المباشرة (إدارة، توثيق، بناء Tera نفسه، مخاطرة)
- إذا تبين أن الهامش لا يغطي التكاليف غير المباشرة → يُرفع إلى 30–40%.

## 10. قواعد الصيانة (Warranty)

- **أول 3 أشهر:** Bug Fix Warranty فقط
  (إصلاح أخطاء برمجية — ليس تطوير ميزات جديدة).
- بعدها: Maintenance Subscription (يحدد لاحقاً).
- **بدون اشتراك صيانة نشط:** أي عمل بعد الضمان يُسعّر لكل حالة بشكل منفصل.

---

## 11. قواعد طلبات التغيير (Change Request)

| النوع | التعريف | التسعير |
|-------|---------|---------|
| **Minor Refinement** | تعديل بسيط لا يغير نطاق/بيانات/Workflow | **مجاني** |
| **Structural Change** | يغير هيكل قاعدة بيانات أو منطق عمل | **مُسعّر** حسب المصفوفة |
| **Scope Increase** | إضافة شاشة أو ميزة أو تكامل جديد | **مُسعّر** حسب المصفوفة |
| **Client-Driven Rework** | إعادة عمل بسبب قرار عميل جديد | **مُسعّر** (وقت فعلي × السعر الداخلي) |
| **Bug Fix** | ضمن الضمان (أول 3 أشهر مجاناً) | مجاني ضمن الضمان |
| **Post-Delivery Change** | بعد التسليم | ضمن عقد الصيانة أو يُسعّر لكل حالة |

## 12. قواعد الاستضافة

- **Tera يوصي فقط** — الزبون يتحمل التكلفة مباشرة.
- لا تشغيل للخوادم أو مسؤولية استضافة حالياً.
- مرفق مع عرض السعر: قائمة بخيارات استضافة موصى بها (اختياري).

---

## 13. عرض السعر للزبون (Client Quote)

- **مبسّط** — يعرض البنود الرئيسية فقط.
- المصفوفة الداخلية لا تُعرض للزبون.
- مثال:

  ```
  Core Web Application:     1,200 JOD
  Inventory Module:           700 JOD
  Reports & Dashboard:        350 JOD
  User Roles:                 250 JOD
  ─────────────────────────────────
  الإجمالي:                 2,500 JOD
  ```

- عرض السعر يتضمن: البنود، الإجمالي، خطة الدفع، صلاحية السعر، العملة، والبنود المستثناة.
- عرض السعر **لا يتضمن**: المصفوفة، معاملات التعقيد، Risk Buffer، الهامش.

## 14. خطة الدفع (Default)

| الدفعة | النسبة | الشرط |
|--------|:-----:|-------|
| 1 | **50%** | بداية التعاقد |
| 2 | **25%** | بعد اعتماد النموذج / المرحلة الأولى |
| 3 | **25%** | عند التسليم النهائي |

للمشاريع الكبيرة: يمكن استخدام خطة بديلة مثل 40%–30%–20%–10% أو حسب الاتفاق.

## 15. صلاحية عرض السعر (Quote Validity)

| الحجم | المدة |
|-------|:-----:|
| افتراضي | **14 يوماً** |
| المشاريع الكبيرة | **30 يوماً** |

بعد انتهاء الصلاحية: يُعاد تقييم السعر (قد يتغير بناءً على المتغيرات الحالية).

TCEA يصدر Draft Quotation فقط.

## 17. العملة والضرائب والرسوم

- جميع الأسعار **بالدينار الأردني (JOD)**.
- الأسعار **لا تشمل**:
  - أي ضرائب أو رسوم حكومية
  - رسوم بوابات الدفع
  - رسوم اشتراكات خارجية (تراخيص، APIs، خدمات طرف ثالث)
  - تكاليف الاستضافة أو الدومين أو SSL
- أي من هذه البنود إذا كانت ضمن النطاق، **تُذكر صراحةً وبشكل منفصل** في عرض السعر.

**مثال:** "السعر لا يشمل ضريبة المبيعات — تُضاف عند الفوترة حسب القانون."

## 18. مراحل إخراج السعر (Pricing Output Levels)

ثلاثة مستويات لإخراج السعر، كل مستوى له صلاحية مختلفة:

| المستوى | المسمى | الصلاحية | متى يصدر |
|---------|--------|---------|----------|
| **Level 1** | تقدير مبدئي (Preliminary Estimate) | **غير ملزم** — نطاق سعري فقط | بعد أول مقابلة |
| **Level 2** | مسودة عرض سعر (Draft Quotation) | **مسودة** — يحتاج اعتماد Majed | بعد توثيق النطاق والميزات |
| **Level 3** | عرض سعر رسمي (Official Quotation) | **ملزم** بعد اعتماد Majed | بعد اعتماد Level 2 |

### Level 1 — Preliminary Estimate

- يُنتج بعد أول مقابلة مع العميل.
- **غير ملزم** — نطاق سعري تقريبي فقط.
- الهدف: تحديد هل ميزانية العميل مناسبة قبل الاستمرار.
- لا يُستخدم كعرض سعر رسمي ولا يُرسل للعميل كعرض.

**مثال:**
> "حسب المعلومات الأولية، المشروع يبدو ضمن نطاق 1,500 إلى 2,500 دينار، لكن السعر النهائي يحتاج تحليل نطاق وتثبيت المتطلبات."

### Level 2 — Draft Quotation

- يُنتج بعد توثيق النطاق والميزات والشاشات الأساسية.
- **مسودة فقط** — يحتاج مراجعة واعتماد Majed.
- يستخدم `TeraPricingPolicy.md` لحساب السعر (معادلة، Rubric، Risk Buffer).
- يُحفظ في `client-engagement/` مع إشارة واضحة: **DRAFT — لم يعتمد بعد**.

**قاعدة جاهزية إلزامية:**

لا يجوز إنتاج `Level 2 Draft Quotation` إلا بعد اجتياز **Quotation Readiness Gate** في TCEA، وبحد أدنى وضوح:

- MVP Scope
- Out of Scope
- Screens estimate
- Reports / Dashboards estimate
- Integrations included / excluded
- Notifications included / excluded
- Design direction
- Technical / Hosting assumption
- Security assumptions
- Commercial risks
- Delivery assumptions

إذا بقي شيء غير واضح، يجب توثيقه صراحة كـ `Assumption` أو `Risk` أو `Deferred`، وليس تجاهله.

### Level 3 — Official Quotation

- **لا يصدر إلا بعد اعتماد Majed الصريح.**
- يكون صالحاً للمدة المحددة في صلاحية عرض السعر (§15).
- يُرسل للعميل كعرض رسمي.

## 19. توقيت الإخراج حسب تصنيف المشروع

قبل إنتاج أي سعر، يُصنّف المشروع إلى واحدة من أربع فئات:

### أ. مشروع صغير واضح
مثل: موقع تعريفي، Landing Page، تطبيق CRUD بسيط، متجر صغير بنطاق معروف.

```
1. Client Intake ← جمع معلومات أولية (أول مقابلة)
2. Preliminary Estimate (Level 1) ← نطاق سعري تقريبي
3. CLIENT_BRIEF + SCOPE_SUMMARY سريع ← ملفان خفيفان
4. Discovery Coverage + Quotation Readiness
5. Draft Quotation (Level 2) ← من TCEA
6. Majed Approval ← اعتماد
7. Official Quotation (Level 3) ← إرسال للعميل
```

### ب. مشروع متوسط
مثل: نظام مستودعات صغير، منصة خدمية مع صلاحيات، تطبيق إدارة مهام.

```
1. Client Intake (أول مقابلة)
2. Preliminary Estimate (Level 1)
3. ملفات نطاق أساسية: CLIENT_BRIEF, SCOPE_AND_BOUNDARIES, FEATURE_LIST
4. Discovery Coverage + Quotation Readiness
5. Draft Quotation (Level 2)
6. Majed Approval
7. Official Quotation (Level 3)
```

### ج. مشروع معقد
مثل: نظام مالي، ERP صغير، منصة تعليمية كاملة، تطبيق موبايل + ويب مع تكاملات.

```
1. Client Intake (أول مقابلة)
2. Preliminary Estimate (Level 1)
3. تحليل نطاق شامل: ملفات SCOPE, MODULES, SCREENS, BUSINESS_RULES
4. Discovery Coverage + Quotation Readiness
5. Draft Quotation (Level 2)
6. Majed Approval
7. Official Quotation (Level 3)
```

### د. مشروع غامض
العميل: "أريد نظام مثل طلبات"، "أريد منصة تعليمية كاملة"، "أريد ERP".

```
1. Client Intake (أول مقابلة)
2. Paid Discovery (§7) ← 50–100 JOD أو 5% — يُخصم عند التعاقد
3. Discovery Output ← CLIENT_BRIEF + SCOPE_SUMMARY
4. Discovery Coverage + Quotation Readiness
5. Draft Quotation (Level 2)
6. Majed Approval
7. Official Quotation (Level 3)
```

### القاعدة الحاسمة للتوقيت

```text
أول مقابلة = تقدير مبدئي فقط (Level 1)
بعد توثيق النطاق + Quotation Readiness = Draft Quotation (Level 2)
بعد اعتماد Majed = Official Quotation (Level 3)
لا يصدر عرض سعر رسمي من أول مقابلة أبداً.
```

## 20. آلية الاحتساب — خطوات TCEA للتقدير

هذه الخطوات يتبعها TCEA عند إنتاج **Draft Quotation (Level 2)**:

```
1. تحليل المتطلبات → تفكيك المشروع إلى ميزات/شاشات
2. لكل ميزة/شاشة:
   a. تقدير Base Price (ساعات Simple × السعر الداخلي)
   b. تطبيق Rubric التعقيد → حساب المعامل
   c. جمع تكاليف التكامل (API, إلخ)
3. تطبيق Risk Buffer (حسب 5 معايير)
4. إضافة هامش الربح (20%)
5. مقارنة الناتج مع Minimum Project Price
6. إنتاج Draft Quotation
```

**ملاحظة:** الـ 15–25 JOD/ساعة سعر داخلي تجريبي — لا يُشار إليه خارج Tera.
