# أفضل الممارسات القياسية — Tera Maintenance

> 🔑 **هذا هو "دستور" المنتج.** كل ما يليه هو المعيار المعتمد من Tera لأنظمة صيانة الأجهزة.
> يُعطى لأي عميل جديد لقراءته قبل النقاش — يخفف عبء العمل ويمنع إعادة اختراع العجلة.
> **المصدر:** البحث المخصص (33 مصدراً) + Domain Intelligence Report — **وليس كلام أي عميل**.

---

## 1. معايير سير العمل (Workflow Standards)

### T-WF-MNT-001 — دورة حياة الطلب (إلزامي ✅ Core)

```
المرحلة 1: الطلب (Request)
    ↓
المرحلة 2: الفرز والتقييم (Triage)
    ↓
المرحلة 3: التعيين (Assign)
    ↓
المرحلة 4: التنفيذ (Execute)
    ↓
المرحلة 5: الإغلاق (Close)
    ↓
المرحلة 6: التحليل (Analyze)
```

| المرحلة | الوصف | المسؤول |
|:--------|:-------|:--------|
| 1. الطلب | استقبال الطلب + تسجيل العميل والجهاز | استقبال |
| 2. الفرز | تحديد الأولوية + تقييم الأهمية | استقبال/مدير |
| 3. التعيين | اختيار الفني المناسب + تحديد الموعد | مدير |
| 4. التنفيذ | الفني يشخّص + يسجّل + يطلب موافقة | فني |
| 5. الإغلاق | تسجيل الدفع + تسليم الجهاز | استقبال/محاسب |
| 6. التحليل | تقرير + قياس أداء | مدير |

### T-WF-MNT-002 — إدارة الحالة (إلزامي ✅ Core)

**الحالات المعيارية (8 حالات كحد أقصى):**

```
1. جديد (New)
2. قيد المراجعة (Under Review)        ← الفرز والتقييم
3. تم التخصيص (Assigned)              ← الفني معيّن
4. قيد التنفيذ (In Progress)          ← الفني يشتغل
5. بانتظار موافقة العميل (Pending Approval)
6. معلق (On Hold)                     ← بانتظار قطعة/معلومة
7. تم الإنجاز (Completed)             ← جاهز للتسليم
8. مغلق (Closed)                      ← سُلّم + سُدِد
```

**القواعد:**
- لا تزيد عن 8 حالات — كثرة الحالات تربك المستخدمين
- كل تغيير حالة = سجل في `StatusHistory` (من، متى، من أي حالة إلى أي حالة)
- إشعار تلقائي للأطراف المعنية (انظر §5)

### T-WF-MNT-003 — موافقة العميل (إلزامي ✅ Core)

```
1. بعد الفحص → تسجيل التكلفة المقترحة
2. حالة الطلب: "بانتظار موافقة العميل"
3. تسجيل طريقة الموافقة: [واتساب / مكالمة / داخل الورشة]
4. تسجيل: هل وافق؟ (نعم/لا) + من سجّل + متى
5. إذا وافق → "قيد التنفيذ"
6. إذا رفض → "مغلق" (أو إرجاع الجهاز)
```

**القاعدة الحاسمة:** لا يبدأ التنفيذ WITHOUT توثيق الموافقة. هذا يحمي المنظومة والعميل قانونياً.

### T-WF-MNT-004 — نظام الأولويات (إلزامي ✅ Core)

| الأولوية | اللون | المعنى | الترتيب في القائمة |
|:--------|:------|:-------|:-------------------|
| **عادي** (Normal) | 🟢 أخضر | طلب روتيني | الأخير |
| **مهم** (Important) | 🟡 أصفر | يحتاج انتباه | وسط |
| **طارئ** (Urgent) | 🔴 أحمر | عميل مستعجل / جهاز أساسي / صيف | **الأول** |

**القاعدة:** الطلبات الطارئة تظهر أعلى القائمة دائماً.

## 2. معايير نموذج البيانات (Data Model Standards)

### T-DM-MNT-001 — الكيانات الأساسية (إلزامي ✅ Core)

```
العميل (Customer)
  ├── له عناوين متعددة
  ├── له أجهزة (Asset)
  ├── له طلبات (WorkOrder)
  └── له مدفوعات

الجهاز (Asset) — كيان مستقل مرتبط بالعميل
  ├── نوع (DeviceType): ثلاجة/غسالة/مكيف/شاشة...
  ├── ماركة (Brand)
  ├── موديل (Model) — اختياري
  ├── رقم سيريال (Serial) — اختياري
  └── تاريخ صيانة (من الطلبات السابقة)

طلب الصيانة (WorkOrder)
  ├── مرتبط بعميل + جهاز
  ├── مرتبط بفني
  ├── له حالة (Status) + أولوية (Priority)
  ├── له تكلفة (مقترحة + فعلية)
  ├── له موافقة (طريقة + حالة + توقيت)
  ├── له مدفوعات
  └── له سجل حالة (StatusHistory)

الفني (Technician)
  ├── مرتبط بـ User
  ├── له مهارات (Skills)
  └── له طلبات مسندة

المستخدم (User)
  ├── له دور (Role): مدير/استقبال/فني/محاسب
  └── مرتبط بفرع (Branch) — حتى لو فارغ

الدفع (Payment)
  └── مرتبط بطلب
```

### T-DM-MNT-002 — تصميم للتوسع (إلزامي ✅ Core)

| المبدأ | التطبيق |
|:-------|:--------|
| **branch_id** | في كل جدول رئيسي — لتصفية البيانات حسب الفرع من اليوم الأول (حتى لو عميل بفرع واحد) |
| **Audit Trail** | جدول `StatusHistory` لكل كيان رئيسي — يمنع الإنكار ويوفر تتبعاً |
| **Soft Delete** | حقل `deleted_at` — لا حذف فيزيائي |
| **JSON fields** | للحقول المتغيرة (مثل المهارات، التفاصيل الإضافية) — مرونة بدون تغيير Schema |
| **Indexing** | على الحقول الأكثر بحثاً: `wo_number`, `phone`, `status`, `date` |
| **العلاقات** | ماركة ← موديل ← قطعة غيار (Many-to-Many) — جاهز للتوسع لاحقاً |

**ملاحظة:** جدول `SparePart` و`Supplier` و`Warehouse` و`ServiceContract` جاهزة من الآن (حتى لو فارغة) — لتجنب إعادة البناء عند التوسع.

## 3. معايير إدارة المخزون (Inventory Standards)

### T-INV-MNT-001 — تصنيف قطع الغيار (قياسي ⚪ Standard — Phase 2)

- **تصنيف ABC:** A (عالية القيمة، قليلة) / B (متوسطة) / C (رخيصة، كثيرة)
- **Criticality Analysis:** حسب أهمية القطعة لاستمرارية الخدمة
- **الربط:** قطعة الغيار ← أنواع الأجهزة/الموديلات (Many-to-Many)

### T-INV-MNT-002 — إعادة الطلب التلقائي (متقدم 🔶 Advanced — Phase 2+)

- **Reorder Point:** حد أدنى للمخزون عند الوصول له ← أمر شراء تلقائي
- **Safety Stock:** مخزون أمان لمواجهة تقلبات الطلب
- **Auto-Deduct:** عند إغلاق أمر عمل ← إنقاص الكمية المستخدمة آلياً
- **Low Stock Alerts:** إشعار عند انخفاض الكمية

## 4. معايير التكامل (Integration Standards)

### T-INT-MNT-001 — تكامل واتساب (قياسي ⚪ Standard — Phase 2)

- **الأداة:** WhatsApp Business Cloud API (وليس واتساب العادي)
- **الاستخدام:** إشعارات، عروض أسعار، موافقة (Interactive Template)، استقبال طلبات
- **الأولوية:** قصوى في السوق العربي — العميل يتوقع واتساب كقناة أولى

### T-INT-MNT-002 — تصميم API للتوسع (قياسي ⚪ Standard — من اليوم الأول)

```
- RESTful + JSON
- OpenAPI/Swagger موثّق
- Rate Limiting
- Webhooks للأحداث (طلب جديد، تغيير حالة، دفع)
- Versioning: /api/v1/, /api/v2/
```

**Endpoints الأساسية:**
```
GET/POST   /api/v1/customers
GET/POST   /api/v1/work-orders
PATCH      /api/v1/work-orders/{id}/status
GET/POST   /api/v1/spare-parts/consume
POST       /api/v1/payments
GET        /api/v1/reports/kpi
POST       /api/v1/notifications/send
```

## 5. معايير الإشعارات (Notification Standards)

### T-NOT-MNT-001 — إشعارات موجهة حسب الدور (إلزامي ✅ Core)

| الدور | يستلم إشعار عند |
|:-----|:----------------|
| **فني** | إسناد طلب جديد + تغيير حالة طلبه |
| **استقبال** | جهاز جاهز للتسليم + يحتاج تواصل مع عميل |
| **مدير** | طلب متأخر + طلب طارئ + طلب بقي طويلاً بلا تحديث |
| **محاسب** | طلب جاهز وعليه مبلغ/دفعة مستحقة |

**القاعدة الذهبية:** فقط الأحداث التي **تحتاج تصرّفاً** من المستلم — لا إزعاج بكل حركة صغيرة.

## 6. معايير تجربة المستخدم (UX/UI Standards)

### T-UX-MNT-001 — Mobile-First + Offline-First (إلزامي ✅ Core)

- التصميم للجوال أولاً (الفنيون في الميدان)
- العمل دون اتصال مع مزامنة لاحقة (مهم للورش والزيارات الميدانية)

### T-UX-MNT-002 — دعم العربية الكامل (إلزامي ✅ Core)

- RTL متكامل
- خطوط تدعم العربية
- دعم الأرقام (هندية/عربية)
- تقويم هجري + ميلادي

### T-UX-MNT-003 — مبادئ لكل دور

| الدور | الأولوية في التصميم |
|:-----|:-------------------|
| **مدير** | لوحة تحكم سريعة + تقارير بيانية |
| **استقبال** | جدولة Drag-and-Drop + تحويل سريع |
| **فني** | 3-4 نقرات كحد أقصى لإنجاز المهمة + أزرار كبيرة |
| **محاسب** | واجهة فواتير ومدفوعات واضحة |
| **عميل** (عبر موظف) | تتبع بسيط برقم الطلب + رقم الهاتف |

## 7. معايير التقارير (Reporting Standards)

### التقارير الأساسية (Level 1 — إلزامي ✅)

| التقرير | المحتوى |
|:--------|:--------|
| **يومي** | عدد الطلبات الجديدة/المنجزة/قيد التنفيذ |
| **أسبوعي** | ملخص المدير: جديد، منجز، متأخرات، إيراد، أكثر فني إنجازاً |
| **شهري** | ملخص الإيرادات + فلترة حسب الفني |
| **المتأخرات** | الطلبات التي تجاوزت المدة المتوقعة |

### تقارير متقدمة (Level 2+ — Standard/Advanced)

- **KPIs:** MTTR (متوسط وقت الإصلاح)، FTFR (نسبة الإصلاح من أول مرة)، PM/CM Ratio
- **تكلفة الصيانة لكل أصل**
- **رضا العملاء (CSAT)**

## 8. معايير البنية التحتية (Architecture Standards)

### T-ARC-MNT-001 — تعدد الفروع (إلزامي ✅ Core)

- كل جدول رئيسي يحمل `branch_id`
- المدير يرى كل الفروع — الفني يرى فرعه فقط
- لا تكلفة إضافية في النسخة الأولى — فقط جاهزية في DB

### T-ARC-MNT-002 — الأمان والامتثال

- تشفير HTTPS إلزامي
- صلاحيات الأدوار تمنع الوصول غير المصرح به
- **PDPL الأردني:** تشفير بيانات العملاء + احترام الخصوصية (استشارة قانونية عند الحاجة)

## 9. قواعد التخصيص (Customization Rules)

عندما يطلب عميل انحرافاً عن هذا المعيار:

| النوع | الإجراء |
|:------|:--------|
| **إضافة ميزة ضمن Level الحالي** | مسموح — تُضاف لملف العميل كـ "تخصيص" |
| **تخفيض ميزة من المعيار** | مسموح — لكن يُوثّق كـ "نطاق مخفض" صراحةً |
| **ميزة من Level أعلى** | تُعامل كـ Phase تالية أو Change Request |
| **تغيير جوهري في Workflow** | يتطلب مراجعة Majed — لا يُعدّل المعيار نفسه |

**القاعدة الذهبية:** المعيار لا يُكسر لرغبة عميل — بل يُوثّق الانحراف في ملف العميل.
