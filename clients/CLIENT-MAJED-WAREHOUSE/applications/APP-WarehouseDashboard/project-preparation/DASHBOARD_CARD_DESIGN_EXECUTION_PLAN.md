## 1) الهدف من هذه الخطة

هذه الخطة لا تهدف فقط إلى "تحسين شكل البطاقات"، بل إلى تحقيق هدفين معاً:

1. **إخراج بصري فاخر ومنظم** للبطاقات داخل الداشبورد، مستوحى من المرجع البصري الذي اعتمده Majed.
2. **الاستفادة الكاملة من كل معطيات البطاقة** المخزنة في الإعدادات، بحيث لا تبقى القيم محفوظة فقط في قاعدة البيانات دون تأثير فعلي على العرض أو التفاعل.

النتيجة المطلوبة في النهاية:

> كل بطاقة في الداشبورد يجب أن تكون **صحيحة وظيفياً** و**مقنعة بصرياً** في آن واحد.

## 2) القيد الصريح المعتمد لهذه الخطة

### ممنوع استخدام Syncfusion في هذا المسار

أي تطوير جديد ضمن هذه الخطة الخاصة بتصميم البطاقات وعرضها وتفاعلها يجب أن يلتزم بالآتي:

- **لا Syncfusion**
- أي مكتبة خارجية مساعدة يجب أن تكون خفيفة وواضحة وتخدم الهدف مباشرة
- لا نضيف مكتبات كثيرة بلا داعٍ

### البدائل المسموح بها مبدئياً

| الاستخدام | المكتبة المفضلة | ملاحظات |
|---|---|---|
| الرسوم البيانية الأساسية | **ApexCharts** | مناسب جداً لـ RTL والبطاقات التنفيذية |
| Tooltip احترافي | **Tippy.js** أو Tooltip مخصص خفيف | للـ Description وما شابهه |
| Drag & Drop | **SortableJS** | مستخدم بالفعل ومناسب |
| الأيقونات | **Lucide** أو SVG Icons محلية | خفيفة وواضحة |
| Micro animations | CSS transitions / Motion خفيف | بدون تضخيم |

> **قرار تنظيمي:** لا نضيف مكتبة جديدة إلا إذا كان لها دور واضح في البطاقة نفسها.

## 3) المرجع البصري المستهدف (Design Direction)

اعتماداً على الصورة المرجعية التي شاركها Majed، اللغة البصرية المستهدفة للبطاقات هي:

- **تنفيذي فاخر**
- **هادئ ونظيف**
- **أزرق / أبيض / ظلال ناعمة**
- **هرمية واضحة جداً للمعلومة**
- **مساحات مريحة**
- **بطاقات كبيرة نسبياً وليست مزدحمة**
- **حالات لونية قليلة ومضبوطة**:
  - أخضر = إيجابي
  - برتقالي = تنبيه
  - أحمر = خطر

### خصائص البطاقة المستهدفة

1. Header نظيف
   - أيقونة داخل مربع أو كبسولة بلون الهوية
   - عنوان البطاقة
   - سطر مساعد/وصف مختصر أو Tooltip

2. Body قوي بصرياً
   - رقم رئيسي كبير أو رسم واضح
   - ترتيب بصري يختلف حسب نوع البطاقة

3. Footer ذكي عند الحاجة
   - معلومة تفسيرية
   - حالة عامة
   - إجمالي عام
   - سطر مقارنة

4. شخصية مختلفة حسب نوع البطاقة
   - KPI بسيط ≠ KPI مع تغيّر ≠ KPI متقدم ≠ بطاقة جدول ≠ بطاقة تنبيهات

## 4) المبدأ الحاكم للتطوير

### لا نعامل إعدادات البطاقة كـ metadata فقط

أي إعداد موجود في `DashboardCard` يجب أن يُصنّف ضمن واحدة من ثلاث فئات فقط:

| التصنيف | المعنى |
|---|---|
| **Visual Driver** | يؤثر مباشرة على شكل البطاقة |
| **Behavior Driver** | يؤثر على التفاعل أو التحديث أو الفلترة |
| **Data Driver** | يؤثر على الاستعلام أو حساب القيمة أو تفسيرها |

وإذا لم نستطع وضع إعداد ما ضمن واحدة من هذه الفئات بشكل فعلي، فإما:

1. نربطه تنفيذاً بشكل صحيح، أو
2. نزيله من تجربة المستخدم، أو
3. نؤجله رسمياً مع توثيق واضح.

## 5) خريطة الإعدادات → كيف يجب أن تُستخدم فعلياً

## 5.1 إعدادات مشتركة لكل البطاقات

| الإعداد | نوعه | كيف يجب أن يُستخدم |
|---|---|---|
| `Title` | Visual | عنوان البطاقة في الهيدر |
| `Description` | Visual | Tooltip أو subtitle أو helper text |
| `ChartType` | Visual/Data | يحدد بنية البطاقة ومحرّك العرض |
| `ColorPalette` | Visual | يحدد ألوان الرسم/الحالة/التفاصيل الثانوية لهذه البطاقة تحديداً |
| `RefreshInterval` | Behavior | مؤقت تحديث مستقل لهذه البطاقة |
| `GridWidth/Height/X/Y` | Visual/Layout | حجم ومكان البطاقة في الشبكة |

## 5.2 إعدادات KPI

| الإعداد | نوعه | الاستخدام المطلوب |
|---|---|---|
| `AggregationType` | Data | تحديد طريقة حساب القيمة الرئيسية |
| `KpiMode` | Visual/Behavior | تحديد بنية البطاقة: بسيط / مع تغيّر / متقدم |
| `ValueColumn` | Data | العمود الرقمي الأساسي |
| `DateColumn` | Data/Behavior | العمود الزمني لتطبيق الفلاتر والمقارنة والرسم |
| `CategoryColumn` | Data/Interaction | Breakdown / Drill / تصنيف بصري حسب القرار النهائي |
| `ShowChange` + `ChangeSource` | Visual/Data | إظهار نسبة التغير واتجاهها مع نص المقارنة |
| `ShowSparkline` + `SparklineMonths` | Visual/Data | رسم اتجاه مصغر للفترة المحددة |
| `ShowGrandTotal` + `GrandTotalSource` | Visual/Data | سطر إجمالي عام أو secondary metric |
| `DateFilterMode` | Behavior/Data | تحديد هل البطاقة تتبع فلتر الداشبورد أم فلترها الداخلي |
| `FixedStartDate/FixedEndDate` | Data | النطاق الثابت عند الحاجة |
| `RelativeDays` | Data | آخر N يوم عند اختيار الفلترة النسبية |

## 6) الأنواع البصرية المستهدفة للبطاقات

### 6.1 KPI بسيط

**العناصر الأساسية:**
- Header: icon + title + description hint
- Main value: رقم كبير جداً
- Unit / label صغير
- بدون ازدحام

**الاستخدام المثالي:**
- إجمالي المخزون
- عدد الطلبات
- عدد الأصناف النشطة

### 6.2 KPI مع تغيّر

**العناصر:**
- كل ما في KPI البسيط
- Change badge (٪ + سهم + لون)
- سطر تفسير: مقارنة بالشهر السابق / الفترة السابقة

**الاستخدام المثالي:**
- الطلبات اليوم
- المبيعات مقابل أمس
- نسبة الإنجاز الحالية

### 6.3 KPI متقدم

**العناصر:**
- الرقم الرئيسي
- change badge
- sparkline
- سطر إجمالي عام أو ملاحظة تفسيرية
- إمكانية footer status

**الاستخدام المثالي:**
- دوران الصنف
- قيمة المخزون خلال فترة
- مؤشر أداء مركب

### 6.4 بطاقة Chart

**العناصر:**
- Header فاخر مثل KPI
- رسم واضح بألوان البطاقة نفسها
- footer summary صغير عند الحاجة

**الأنواع:**
- Bar
- Line
- Pie
- Gauge

### 6.5 بطاقة Table / Alerts / Summary

**العناصر:**
- عنوان + أيقونة + action صغير
- صفوف قليلة وواضحة
- حالات لونية مدروسة
- CTA خفيف في الأسفل إن لزم

## 7) الفجوات الحالية التي تعالجها هذه الخطة

هذه الخطة تعالج بشكل مباشر الفجوات الموثقة في:
`CARD_SETTINGS_UTILIZATION_ANALYSIS_AND_PLAN.md`

### الفجوات ذات الأولوية العليا

1. `DateFilterMode` لا يؤثر فعلياً على نتائج البطاقة
2. `preset` القادم من شريط التواريخ في الداشبورد يُرسل من الواجهة ويُهمل في الخادم
3. `RefreshInterval` محفوظ لكنه لا يفعّل تحديثاً تلقائياً
4. `ColorPalette` محفوظة لكن لا تُستخدم فعلياً لكل بطاقة
5. `Description` محفوظ لكنه لا يظهر

### الفجوات ذات الأولوية المتوسطة

6. `CategoryColumn` غير مستثمر فعلياً
7. `GrandTotalSource` نظري أكثر من عملي حالياً
8. معاينة الـ Builder لا تعكس كل سلوك البطاقة النهائية

## 8) الترتيب الصحيح للتنفيذ

### Phase A — تثبيت القاعدة البصرية للبطاقات

**الهدف:** توحيد الشكل العام للبطاقات قبل توسيع السلوك

يشمل:
- Card shell الموحد
- Header pattern
- KPI simple / with change / advanced visual states
- Chart card shell
- Table / alerts shell
- استعمال `Description`
- استعمال `ColorPalette`

**المخرجات المتوقعة:**
- شكل بصري واضح موحّد لكل أنواع البطاقات
- عدم وجود بطاقة "صحيحة وظيفياً لكنها فقيرة بصرياً"

---

### Phase B — ربط الإعدادات بالسلوك الفعلي

**الهدف:** جعل كل إعداد له تأثير حقيقي على البطاقة

يشمل:
- `RefreshInterval` per-card auto refresh
- `DateFilterMode` الحقيقي
- `FixedStartDate/FixedEndDate/RelativeDays`
- احترام فلتر الداشبورد الزمني
- ربط الاستعلامات بطريقة آمنة وواضحة

**هذه المرحلة هي الأهم** لأنها تمس صحة الأرقام المعروضة.

---

### Phase C — تعميق تفاعل بطاقات KPI

**الهدف:** استثمار حقول KPI المتقدمة بالكامل

يشمل:
- تحسين change presentation
- تحسين sparkline rendering
- تفعيل `GrandTotalSource` حسب القرار
- استثمار `CategoryColumn` بالشكل الذي يعتمده Majed

---

### Phase D — اتساق تجربة الإدارة والبناء

**الهدف:** تقليل الفجوة بين ما يراه الأدمن في الـ Builder وبين البطاقة النهائية في الداشبورد

يشمل:
- تحسين preview behavior
- مواءمة المخرجات البصرية مع التنفيذ النهائي

## 11) معايير القبول النهائية

لن نعتبر هذا المسار ناجحاً إلا إذا تحققت المعايير الآتية:

### بصرياً
- كل البطاقات تشترك في لغة تصميم موحدة وراقية
- KPI simple / with change / advanced تبدو مختلفة وظيفياً وبصرياً
- الرسم البياني والـ table والـ alert cards كلها تحمل نفس الهوية التنفيذية الهادئة
- لا توجد بطاقة تبدو placeholder أو تقنية أكثر من اللازم

### وظيفياً
- `Description` يظهر فعلياً في البطاقة
- `ColorPalette` يغيّر ألوان البطاقة فعلاً
- `RefreshInterval` يحدث البطاقة تلقائياً فعلاً
- `DateFilterMode` يعمل فعلياً حسب وضعه
- فلتر التاريخ في الداشبورد يؤثر فعلياً على البطاقات المستهدفة
- `SparklineMonths` يغيّر عدد الأشهر فعلياً
- `ShowGrandTotal` و `ShowChange` و `ShowSparkline` تؤثر على العرض الحقيقي وليس الحفظ فقط

### هندسياً
- لا Syncfusion في هذا المسار
- البناء ينجح بدون أخطاء
- لا يوجد تضخيم غير ضروري في المكتبات
- كل تغيير قابل للاختبار بشكل مستقل

## 12) ما الذي نحتاجه من Majed قبل التنفيذ

لكي نبدأ التنفيذ بدقة، نحتاج تثبيت القرارات التالية:

1. **هل فلتر التاريخ في الداشبورد يخص KPI فقط أم كل أنواع البطاقات؟**
2. **ما الشكل المطلوب لاستثمار `CategoryColumn`؟**
   - Breakdown بصري؟
   - Drill-down تلقائي؟
   - ملخص تفصيلي صغير؟
3. **هل `GrandTotalSource` نطوره بالكامل الآن أم نثبت نسخة `sameTable` فقط في هذه المرحلة؟**
4. **هل تريد Tooltip بسيط للوصف أم subtitle ظاهر دائماً أم الاثنين معاً؟**
5. **هل auto refresh يكون صامتاً أم مع مؤشر بصري خفيف؟**
