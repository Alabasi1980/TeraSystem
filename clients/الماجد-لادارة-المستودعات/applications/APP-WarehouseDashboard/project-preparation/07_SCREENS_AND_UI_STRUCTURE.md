## 1. نظرة عامة (Overview)

تُعرّف هذه الوثيقة جميع شاشات الواجهة وبنية التخطيط لتطبيق **WarehouseDashboard**، وهي نظام محلي يعمل على IIS لعرض بيانات المستودعات في Dashboard ديناميكي، مع لوحة إدارة (Admin Panel) لإدارة البطاقات، وشاشة سجلات المزامنة.

الواجهة مبنية بـ **ASP.NET Core Razor Pages** بدون SPA، وتستخدم مكتبة **Syncfusion Essential Studio** (عبر حزمة `Syncfusion.EJ2.AspNet.Core` — NuGet) لعرض الرسوم البيانية والجداول والمؤشرات.

تُقسَّم الشاشات إلى ثلاث مجموعات:

| المجموعة | الشاشات | الوصول |
|---------|---------|--------|
| **Dashboard (عرض البيانات)** | Dashboard Main، Drill Down | مفتوح للشبكة المحلية (بدون مصادقة في Phase 1) |
| **Admin Panel (إدارة التكوين)** | Admin Login، Card List، Card Editor، Query Tester، Drill Down Config | محمي بكلمة مرور + رابط مخفي `/admin-secure-panel` |
| **Sync Logs (سجلات المزامنة)** | Sync Logs | متاح لأدمن (عبر Dashboard أو Admin) |

## 2. مبادئ التصميم الثابتة (Design Principles)

هذه المبادئ ملزمة لكل شاشة ويجب أن تُطبَّق بدقة:

### 2.1 الهوية البصرية — الموضوع الأزرق (11 لوناً)
- التطبيق بالكامل يطبّق لوحة ألوان زرقاء محددة (11 لوناً) معتمدة في `CLIENT_DECISION_LOG.md #9`.
- **المصدر الموحَّد للألوان:** ملف `wwwroot/css/blue-theme.css` (المُعرَّف في `08_TECHNICAL_ARCHITECTURE.md §3.1`) هو المرجع الوحيد للألوان — لا تُعتمد ألوان خارج اللوحة.
- اللوحة تشمل: لون أساسي (Primary)، لون ثانوي (Secondary)، لون تمييزي (Accent)، درجات رمادية زرقاء (Neutrals)، وألوان دلالية (Success/Info/Warning/Danger) ضمن نطاق الأزرق.
- كل مكوّن Syncfusion يُنسَّق عبر Theme مخصص مبني على هذه اللوحة (يمكن توليده عبر Syncfusion Theme Studio وربطه في `_Layout.cshtml`).

### 2.2 الاتجاه من اليمين لليسار (RTL)
- الواجهة عربية افتراضياً؛ الاتجاه `dir="rtl"` في `<html>` عبر `_Layout.cshtml`.
- المحاذاة الافتراضية لليمين؛ القوائم والأزرار الرئيسية على اليمين، ومؤشرات الحالة على اليسار (أو حسب هرمية العناصر).
- الرسوم البيانية (Charts) تتطلب ضبط اتجاه المحاور والـ Legend ليتماشى مع RTL عند الحاجة (مثل عكس ترتيب الفئات).

### 2.3 التخطيط المتجاوب (Responsive)
- نظام شبكي (12 عمود منطقياً) يتكيّف: Desktop (≥1200px)، Tablet (768–1199px)، Mobile (<768px).
- بطاقات Dashboard تتدفّق من عمودين/ثلاثة على الشاشات الكبيرة إلى عمود واحد على الجوال.

### 2.4 مبدأ "أقل لكن أفضل" (Less but Better)
- عدم إضافة عناصر زخرفية لا تخدم قراءة البيانات.
- التباين واللون أهم من الحجم لإبراز العناصر المهمة.
- المساحات البيضاء (Whitespace) تخلق الإحساس بالفخامة والمهنية.

## 3. المكونات المشتركة عبر التطبيق (Shared Components)

تُستخدم هذه المكونات في أكثر من شاشة، ويُنصح ببنائها كـ **Partial Views** مشتركة:

| المكوّن | النوع | الأداة | الشرح |
|---------|-------|--------|-------|
| **شريط حالة المزامنة (Sync Status Bar)** | Partial: `_SyncStatusPartial.cshtml` | HTML + `ejs-progressbar` (أثناء التشغيل) | يعرض آخر وقت مزامنة + مؤشر الحالة (متصل/غير متصل/قيد التشغيل) + زر المزامنة اليدوية |
| **شريط التنقل الهرمي (Breadcrumb)** | Partial: `_BreadcrumbPartial.cshtml` | `ejs-breadcrumb` | يعرض مسار Drill Down ويسمح بالعودة لأي مستوى |
| **إطار البطاقة (Card Container)** | CSS Component | CSS Grid + `blue-theme.css` | حاوية موحّدة للعنوان + المحتوى + الحالة |
| **مؤشر التحميل (Skeleton Loader)** | Component | `ejs-skeleton` | يُعرض أثناء تحميل البيانات لكل بطاقة/جدول/رسم |
| **رسالة الحالة الفارغة (Empty State)** | Component | HTML + Icon | تُعرض عند غياب البيانات مع إبقاء إطار البطاقة |
| **رسالة الخطأ (Error State)** | Component | HTML + Icon | تُعرض داخل البطاقة مع زر "إعادة المحاولة" |
| **إشعارات (Toast)** | Component | Syncfusion Toast (اختياري) | لإبلاغ نجاح/فشل عمليات الحفظ أو المزامنة |

## 4. خريطة التنقل (Navigation Map)

```
الشبكة المحلية (LAN)
│
├── /                              → Dashboard Main (عرض ~20 بطاقة)
│       │
│       └── /Dashboard/DrillDown?cardId=X&level=Y&params=Z
│                                     → Drill Down (مستويات متعددة)
│
├── /SyncLogs/Index                → سجلات المزامنة (جدول)
│
└── /admin-secure-panel/           → (رابط مخفي — غير ظاهر في القائمة)
        │
        ├── /admin-secure-panel/Login            → تسجيل الدخول بكلمة المرور
        ├── /admin-secure-panel/DashboardCards   → قائمة البطاقات (CRUD)
        ├── /admin-secure-panel/CardEditor       → إنشاء/تعديل بطاقة
        ├── /admin-secure-panel/CardEditor/{id}  → تعديل بطاقة موجودة
        ├── /admin-secure-panel/QueryTester      → اختبار استعلام SQL
        └── /admin-secure-panel/DrillDownConfig/{cardId}
                                                → تكوين مستويات Drill Down
```

> **ملاحظة أمنية:**Dashboard و Sync Logs مفتوحة على الشبكة المحلية في Phase 1. Admin Panel محمي بـ `AdminAuthMiddleware` + كلمة مرور BCrypt + جلسة (Session). أي وصول لـ `/admin-secure-panel/*` بدون جلسة صالحة → إعادة توجيه لـ Login.

## 5. الشاشات المطلوبة (Required Screens)

لكل شاشة: الغرض، العناصر الأساسية، الإجراءات، نهج التخطيط، ملاحظات RTL.

### 5.1 Dashboard Main — `/` (أو `/Dashboard/Index`)

**الغرض (Purpose):**
الشاشة الرئيسية التي تعرض شبكة من ~20 بطاقة ديناميكية مبنية آلياً من جدول `DashboardCards` في SQL Server، مع شريط حالة المزامنة وشريط تنقّل (Breadcrumb) في المستوى الأعلى.

**العناصر الأساسية (Key UI Elements):**
- **شريط حالة المزامنة (أعلى الصفحة):** آخر وقت مزامنة، مؤشر الحالة (متصل/قيد التشغيل/خطأ)، زر "مزامنة الآن" (Manual Sync → `POST /api/sync/trigger`).
- **Breadcrumb (مستوى أعلى):** "لوحة المعلومات" فقط (لا يوجد مستوى سابق).
- **شبكة البطاقات (Cards Grid):** ~20 بطاقة، كل بطاقة تُرسَم حسب `ChartType`:
  - `Bar` → `ejs-chart` (Column/Bar)
  - `Line` → `ejs-chart` (Line)
  - `Pie` → `ejs-chart` (Pie)
  - `KPI` → بطاقة رقمية مخصصة + `ejs-sparkline` للمؤشر
  - `Table` → `ejs-grid` (أو جدول HTML بسيط)
  - `Gauge` → `ejs-circulargauge`
- **عنوان البطاقة** + أيقونة صغيرة + (اختيارياً) رابط "تفاصيل" للبطاقات التي لها Drill Down.
- **حالات كل بطاقة:** Skeleton أثناء التحميل، Empty State عند غياب بيانات، Error State عند فشل الاستعلام.

**الإجراءات (Actions):**
- النقر على عنصر داخل بطاقة (عمود/شريحة/صف) له Drill Down مُهيّأ → الانتقال إلى `/Dashboard/DrillDown`.
- النقر على "مزامنة الآن" → تفعيل المزامنة اليدوية وعرض مؤشر التحميل في شريط الحالة.
- تحديث الصفحة يُعيد قراءة آخر البيانات المتزامنة.

**نهج التخطيط (Layout Approach):**
- `_Layout.cshtml` يحوي شريط الحالة أعلى الصفحة دائماً.
- أسفله `main` يحوي Breadcrumb ثم شبكة البطاقات (CSS Grid من 12 عمود منطقي).
- كل بطاقة تُموضَع عبر `GridPositionX/Y` و`GridWidth/Height` من التكوين (انظر §6 أدناه).

**ملاحظات RTL:**
- الشبكة تبدأ من اليمين؛ البطاقة الأولى أقصى اليمين.
- عناوين البطاقات محاذاة لليمين، والقيم الرقمية محاذاة لليمين أيضاً (الأرقام تبقى LTR داخل النص العربي).
- محاور `ejs-chart` تُضبط لتتماشى مع RTL (مثل `enableRtl` حيث يدعمه المكوّن، أو عكس ترتيب الفئات يدوياً).
- شريط الحالة: مؤشر الحالة على اليسار، زر المزامنة على اليمين.

### 5.2 Drill Down — `/Dashboard/DrillDown`

**الغرض (Purpose):**
عرض نتيجة استعلام معامل (parameterized query) لمستوى معيّن من مستويات التعمّق، مع Breadcrumb يعكس المسار الكامل ويسمح بالعودة لأي مستوى سابق.

**العناصر الأساسية (Key UI Elements):**
- **Breadcrumb كامل:** "لوحة المعلومات › [اسم البطاقة الأم] › [المستوى 1] › [المستوى 2]…" — كل عنصر قابل للنقر للعودة.
- **عنوان المستوى الحالي:** اسم `DisplayName` من `CardDrillDownLevels`.
- **الرسم/الجدول الناتج:** يُرسَم حسب `TargetChartType` لنفس المستوى (Bar/Line/Pie/KPI/Table/Gauge).
- **شريط حالة المزامنة:** يبقى أعلى الصفحة (مشترك).
- **حالات:** Skeleton/Empty/Error كما في Dashboard Main.

**الإجراءات (Actions):**
- النقر على عنصر له مستوى Drill Down أعمق → الانتقال لمستوى تالٍ (إضافة معامل جديد للرابط).
- النقر على أي عنصر في Breadcrumb → القفز للمستوى المقابل مباشرة.
- زر "رجوع" (اختياري) بجانب Breadcrumb.

**نهج التخطيط (Layout Approach):**
- نفس الهيكل (Shared Layout) لكن المحتوى بطاقة واحدة/مجموعة بطاقات تعتمد على `DrillDownQuery` مع المعاملات.
- المعاملات تُمرَّر عبر Query String (`cardId`, `level`, `params`) كما في `08_TECHNICAL_ARCHITECTURE.md §6.4`.

**ملاحظات RTL:**
- Breadcrumb يُقرأ من اليمين لليسار: الجذر (لوحة المعلومات) على اليمين، والمستوى الحالي على اليسار — أو يُعرض بفاصل `›` مع اتجاه RTL صحيح.
- الفواصل بين مستويات Breadcrumb تُعكس بصرياً لتتماشى مع اتجاه القراءة.

### 5.3 Admin Login — `/admin-secure-panel/Login`

**الغرض (Purpose):**
صفحة دخول محمية بكلمة مرور لوحة الإدارة (كلمة واحدة مُشفّرة بـ BCrypt). لا تظهر في أي قائمة تنقّل.

**العناصر الأساسية (Key UI Elements):**
- حقل كلمة المرور (Password) — مخفي الحروف.
- زر "دخول".
- رسالة خطأ عند كلمة خاطئة.
- شعار/عنوان بسيط (هوية أزرق).
- (اختيارياً) Anti-Forgery Token مخفي.

**الإجراءات (Actions):**
- إدخال كلمة المرور → `AdminAuthService.VerifyPassword` → عند الصحة: إنشاء جلسة (Session/Cookie) وإعادة توجيه لقائمة البطاقات.
- عند الخطأ: عرض رسالة + السماح بإعادة المحاولة.

**نهج التخطيط (Layout Approach):**
- صفحة مركزية (Centered Card) بسيطة، بدون القائمة الجانبية للـ Admin.
- تستخدم `_Layout` مبسّط أو Layout منفصل للدخول.

**ملاحظات RTL:**
- الحقل محاذاة لليمين، تسمية "كلمة المرور" فوق الحقل أو يساره حسب التصميم.
- زر الدخول يتموضع أسفل الحقل ويتم محاذاته لليمين.

### 5.4 Card List — `/admin-secure-panel/DashboardCards`

**الغرض (Purpose):**
عرض جدول بكل البطاقات الموجودة في `DashboardCards` مع خيارات التعديل والحذف، وأزرار لإنشاء بطاقة جديدة وربط التكوين.

**العناصر الأساسية (Key UI Elements):**
- **جدول البطاقات** (`ejs-grid` أو جدول HTML): الأعمدة: العنوان، نوع الرسم (ChartType)، نوع المصدر (DataSourceType)، الموقع (X/Y)، الحجم (W×H)، الحالة (IsActive)، الإجراءات.
- **أزرار الإجراءات لكل صف:** "تعديل" (→ Card Editor)، "حذف" (مع تأكيد)، (اختيارياً) "تكوين Drill Down" (→ DrillDownConfig).
- **زر "بطاقة جديدة"** أعلى الصفحة.
- **فلتر/بحث** (اختياري) عن طريق العنوان.

**الإجراءات (Actions):**
- "بطاقة جديدة" → `/admin-secure-panel/CardEditor`.
- "تعديل" → `/admin-secure-panel/CardEditor/{id}`.
- "حذف" → تأكيد ثم حذف السجل (مع مستويات Drill Down التابعة إن وُجدت).
- "تكوين Drill Down" → `/admin-secure-panel/DrillDownConfig/{cardId}`.

**نهج التخطيط (Layout Approach):**
- صفحة Admin قياسية: Header للوحة الإدارة + قائمة جانبية/علوية للتنقّل بين صفحات Admin + المحتوى الرئيسي (الجدول).
- الجدول يدعم التصفح والفرز (يدعمه `ejs-grid`).

**ملاحظات RTL:**
- أعمدة الجدول تُرتَّب من اليمين لليسار؛ عمود الإجراءات على اليسار أو اليمين حسب الهرمية (يُنصح بوضعه في أقصى اليسار لسهولة الوصول بـ RTL).
- نصوص الأزرار عربية، الأيقونات تُعكس اتجاهياً عند الحاجة.

### 5.5 Card Editor — `/admin-secure-panel/CardEditor` (و `/{id}`)

**الغرض (Purpose):**
نموذج إنشاء أو تعديل بطاقة واحدة: العنوان، نوع الرسم، مصدر البيانات، الاستعلام، الموقع في الشبكة، الحجم، فترة التحديث، والتفعيل.

**العناصر الأساسية (Key UI Elements) — الحقول (من `08_TECHNICAL_ARCHITECTURE.md §7.4`):**
- **Title** (نص) — عنوان البطاقة.
- **ChartType** (قائمة منسدلة): Bar, Line, Pie, KPI, Table, Gauge.
- **DataSourceType** (قائمة منسدلة): SQL Query / View.
- **SqlQuery** (منطقة نص متعددة الأسطر) — استعلام SQL أو اسم View.
- **GridPositionX / GridPositionY** (رقم) — موقع البطاقة في الشبكة.
- **GridWidth / GridHeight** (رقم) — العرض (1–12) والارتفاع (1–6).
- **RefreshInterval** (رقم) — فترة التحديث التلقائي (ثوانٍ، 0 = بلا).
- **IsActive** (خانة اختيار) — تفعيل/تعطيل البطاقة.
- **أزرار:** "حفظ"، "إلغاء"، (وصلاً بـ Query Tester) "اختبار الاستعلام".

**الإجراءات (Actions):**
- "حفظ" → التحقق من الحقول → `CardConfigService` يحفظ في `DashboardCards` → إعادة توجيه لقائمة البطاقات + Toast نجاح.
- "إلغاء" → رجوع لقائمة البطاقات دون حفظ.
- "اختبار الاستعلام" → فتح Query Tester مُعبّأ مسبقاً بالاستعلام الحالي.

**نهج التخطيط (Layout Approach):**
- نموذج بعمودين على Desktop (الحقول النصية/الرقمية في عمود، ومعاينة البطاقة اختيارية في العمود الآخر)، وعمود واحد على الجوال.
- استخدام Validation summaries واضحة (أخطاء تحت الحقول).

**ملاحظات RTL:**
- تسميات الحقول محاذاة لليمين فوق الحقول.
- القوائم المنسدلة تفتح لأسفل مع نص عربي.
- منطقة `SqlQuery` تحافظ على اتجاه LTR للكود (الاستعلام بالإنجليزية) مع تسمية عربية.

### 5.6 Query Tester — `/admin-secure-panel/QueryTester`

**الغرض (Purpose):**
صفحة لكتابة واختبار استعلام SQL/View قبل حفظه في بطاقة، مع معاينة نتيجة التنفيذ (عدد الصفوف، المدة، جدول النتائج).

**العناصر الأساسية (Key UI Elements):**
- **محرّر الاستعلام** (TextArea LTR) لإدخال SQL.
- **زر "اختبار الاستعلام"** — ينفّذ الاستعلام ضد SQL Server ويعيد النتيجة.
- **منطقة النتيجة:** رسالة الحالة ("✅ 5 صفوف في 152ms") + جدول المعاينة (`ejs-grid` محدود بعدد صفوف للمعاينة).
- **زر "حفظ كـ بطاقة"** (اختياري) ينقل لـ Card Editor مع تعبئة الاستعلام.
- دعم معاملات (Parameters) اختياري لمحاكاة Drill Down.

**الإجراءات (Actions):**
- "اختبار" → تنفيذ الاستعلام وعرض النتيجة/الخطأ.
- "حفظ كـ بطاقة" → تمرير الاستعلام لـ Card Editor.
- "مسح" → تفريغ المحرّر.

**نهج التخطيط (Layout Approach):**
- تقسيم عمودي: محرّر الاستعلام أعلى، نتيجة المعاينة أسفله (أو جانباً على الشاشات الكبيرة).
- الحد من عدد صفوف المعاينة (مثلاً 100 صف) لحماية الأداء.

**ملاحظات RTL:**
- محرّر SQL يبقى LTR (الكود بالإنجليزية)، لكن التسميات العربية حوله RTL.
- جدول النتائج يُرتَّب RTL؛ الأرقام تُعرض بشكل أرقام لاتينية ضمن خلايا RTL.

### 5.7 Drill Down Config — `/admin-secure-panel/DrillDownConfig/{cardId}`

**الغرض (Purpose):**
تكوين مستويات الـ Drill Down لبطاقة محددة: اسم كل مستوى، الاستعلام المعامل الخاص به، ونوع الرسم في ذلك المستوى، والربط بين المستويات.

**العناصر الأساسية (Key UI Elements):**
- **عنوان البطاقة الأم** (للتوضيح).
- **قائمة المستويات** (مرتّبة حسب `Level`): لكل مستوى:
  - `DisplayName` (اسم المستوى — عربي).
  - `DrillDownQuery` (استعلام معامل، LTR).
  - `TargetChartType` (نوع الرسم في المستوى).
  - أزرار تعديل/حذف لكل مستوى.
- **زر "إضافة مستوى"** — يضيف صفاً جديداً في `CardDrillDownLevels`.
- **زر "حفظ التكوين"**.

**الإجراءات (Actions):**
- إضافة/تعديل/حذف مستوى → تحديث `CardDrillDownLevels` مرتبط بـ `ParentCardId`.
- "حفظ" → حفظ جميع المستويات + Toast نجاح.
- (اختيارياً) زر "معاينة Drill Down" يفتح `/Dashboard/DrillDown` بالمعاملات التجريبية.

**نهج التخطيط (Layout Approach):**
- قائمة مستويات عمودية؛ كل مستوى بطاقة فرعية قابلة للطي (Collapsible).
- رقم المستوى يُعرض كـ Badge على اليمين.

**ملاحظات RTL:**
- أرقام المستويات تُعرض كـ Badge على اليمين؛ تفاصيل المستوى على اليسار.
- الاستعلامات تبقى LTR داخل حقولها.

### 5.8 Sync Logs — `/SyncLogs/Index`

**الغرض (Purpose):**
عرض سجلات عمليات المزامنة من جدول `SyncLogs`: وقت البدء/الانتهاء، الحالة، عدد السجلات، المدة، نوع التشغيل (Auto/Manual)، ورسالة الخطأ عند الفشل.

**العناصر الأساسية (Key UI Elements):**
- **جدول السجلات** (`ejs-grid`): الأعمدة: وقت البدء، وقت الانتهاء، الحالة (شارة ملوّنة)، عدد السجلات، المدة (ms)، نوع التشغيل، رسالة الخطأ.
- **فلترة/بحث** (اختياري): حسب الحالة أو التاريخ.
- **تحديث** (يدوي) لجلب أحدث السجلات.
- رابط للوحة الرئيسية.

**الإجراءات (Actions):**
- عرض تفاصيل سجل (عند الفشل: رسالة الخطأ الكاملة).
- تصفية حسب الحالة (Success/Failed/Running).
- (صلاحية Admin فقط في Phase 1).

**نهج التخطيط (Layout Approach):**
- صفحة قياسية بمحتوى جدول كامل العرض.
- شريط الحالة المشترك أعلى الصفحة (للوصول السريع للمزامنة).

**ملاحظات RTL:**
- أعمدة الجدول من اليمين لليسار؛ عمود "الحالة" يُمثَّل بـ Badge ملوّن (أخضر=ناجح، أحمر=فاشل، أزرق=قيد التشغيل) ضمن اللوحة الزرقاء الدلالية.
- التواريخ بصيغة عربية محلية حيث المناسب.

## 6. نظام تخطيط البطاقات (Card Grid System)

كما وُضِّح في `08_TECHNICAL_ARCHITECTURE.md §6.1` و`§3.2`، كل بطاقة لها:

| الحقل | النطاق | الوظيفة في التخطيط |
|-------|--------|---------------------|
| `GridPositionX` | عدد صحيح | العمود المنطقي (1 = أقصى اليمين في RTL) |
| `GridPositionY` | عدد صحيح | الصف المنطقي |
| `GridWidth` | 1–12 | عدد الأعمدة المشغولة |
| `GridHeight` | 1–6 | عدد صفوف الارتفاع |

**النهج:**
- شبكة CSS Grid بـ 12 عمود منطقي؛ في RTL يُعكس ترتيب الأعمدة تلقائياً.
- البطاقات تُرتَّب حسب `GridPositionY` ثم `GridPositionX`.
- البطاقات غير المُموضَعة صراحة تتدفّق تلقائياً (Auto-flow) لتجنّب الفراغات.
- الارتفاع الأدنى للبطاقة يضمن مساحة كافية للرسم + العنوان + الحالة.

## 7. حالات الواجهة (UI States / Vitality)

لكل شاشة تحتوي بيانات، يجب معالجة الحالات التالية (من `08_TECHNICAL_ARCHITECTURE.md §6.5`):

| الحالة | السلوك المطلوب | المكوّن |
|--------|----------------|---------|
| **Loading** | عرض Skeleton لكل بطاقة/جدول/رسم أثناء التحميل | `ejs-skeleton` |
| **Empty** | رسالة "لا توجد بيانات" مع إبقاء إطار البطاقة | Empty State Component |
| **Error** | رسالة الخطأ داخل البطاقة + زر "إعادة المحاولة" | Error State Component |
| **Success** | عرض البطاقة بشكل طبيعي | — |
| **Sync Running** | شريط الحالة: "جاري المزامنة…" + تعطيل زر المزامنة | `ejs-progressbar` + تعطيل الزر |

> **قاعدة الحيوية (Vitality Gate):** أي بروتوتايب/شاشة تُسلَّم يجب أن تبدو "حية" لا "ساكنة" — أي أن تحتوي Skeleton Loading، Empty States، معالجة الأخطاء، وإشعارات (Toast) لنتيجة العمليات، وبيانات تجريبية واقعية المظهر (أسماء مستودعات، أرقام مخزون، مناطق) لا بيانات وهمية فارغة.

## 8. ملاحظات RTL شاملة (Global RTL Notes)

1. **الاتجاه:** `<html dir="rtl" lang="ar">` في `_Layout.cshtml`.
2. **المحاذاة:** النصوص العربية لليمين؛ الأرقام والكود (SQL) تبقى LTR داخل سياقها.
3. **الرسوم البيانية:** ضبط `ejs-chart` لمحاذاة RTL (عكس ترتيب الفئات، محاذاة Legend لليمين، اتجاه المحور حسب نوع البيانات).
4. **الجداول:** الأعمدة من اليمين لليسار؛ عمود الإجراءات يُفضّل في أقصى اليسار لسهولة الوصول.
5. **النماذج:** التسميات فوق الحقول أو يسارها، محاذاة لليمين؛ الحقول الرقمية/النصية محاذاة لليمين.
6. **الأيقونات:** أيقونات الاتجاه (أسهم، رجوع) تُعكس لتتماشى مع RTL.
7. **Breadcrumb:** الجذر على اليمين، والمستوى الحالي على اليسار، مع فواصل ملائمة.
