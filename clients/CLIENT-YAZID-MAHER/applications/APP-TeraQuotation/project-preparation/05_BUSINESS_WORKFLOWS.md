## 2. State Machine — حالات عرض السعر

### 2.1 حالات Status الخمس

```
     ┌────────────────────────────────────────────────────────┐
     │                                                         │
     │  ┌─────────┐                                            │
     │  │  Draft  │  ← مسودة (حفظ أولي، لم تُدخل الأسعار بعد)  │
     │  └────┬────┘                                            │
     │       │                                                 │
     │       ▼                                                 │
     │  ┌──────────────────┐                                   │
     │  │ UpdatedWithPrices│  ← تم إدخال الأسعار، جاهز للطباعة │
     │  └────────┬─────────┘                                   │
     │            │                                            │
     │     ┌──────┴──────┐                                    │
     │     ▼              ▼                                    │
     │  ┌────────┐  ┌────────────┐                             │
     │  │ Printed│  │ PDFExported│                             │
     │  └────────┘  └────────────┘                             │
     │     │              │                                     │
     │     └──────┬───────┘                                    │
     │            ▼                                             │
     │  ┌────────────────┐                                     │
     │  │ SentViaOutlook │  ← إرسال عبر البريد (اختياري)       │
     │  └────────────────┘                                     │
     └────────────────────────────────────────────────────────┘
```

### 2.2 تعريف كل حالة

| الحالة (Status) | القيمة في DB | الوصف | متى تصل؟ |
|:----------------|:------------|:-------|:---------|
| **Draft** | `Draft` | مسودة — العرض قيد الإنشاء أو التعديل. الأسعار قد تكون موجودة أو لا. | عند الحفظ الأولي أو فتح عرض موجود للتعديل |
| **UpdatedWithPrices** | `UpdatedWithPrices` | تم إدخال الأسعار واكتمال البيانات. جاهز للطباعة النهائية | عند الضغط على "طباعة نهائية" لأول مرة أو عند الحفظ مع أسعار كاملة |
| **Printed** | `Printed` | تمت طباعة العرض بشكل نهائي | بعد نجاح عملية الطباعة النهائية |
| **PDFExported** | `PDFExported` | تم تصدير العرض إلى PDF | بعد نجاح تصدير PDF |
| **SentViaOutlook** | `SentViaOutlook` | تم إرسال العرض عبر Outlook | بعد فتح Outlook مع المرفق (لا يُنتظر الإرسال الفعلي) |

### 2.3 خصائص الحالات

| الخاصية | Draft | UpdatedWithPrices | Printed | PDFExported | SentViaOutlook |
|:--------|:-----:|:-----------------:|:-------:|:-----------:|:--------------:|
| **قابل للتعديل** | ✅ | ✅ | ❌ (Read-Only) | ❌ (Read-Only) | ❌ (Read-Only) |
| **قابل للطباعة بدون أسعار** | ✅ | ✅ | ✅ | ✅ | ✅ |
| **قابل للطباعة نهائي** | ❌ | ✅ | ✅ | ✅ | ✅ |
| **قابل للتصدير PDF** | ❌ (ينصح بالطباعة أولاً) | ✅ | ✅ | ✅ | ✅ |
| **قابل للإرسال Outlook** | ❌ | ✅ | ✅ | ✅ | ✅ |
| **قابل للحذف** | ✅ | ✅ | ✅ (مع تأكيد إضافي) | ✅ (مع تأكيد إضافي) | ✅ (مع تأكيد إضافي) |

> **تنبيه:** التعديل مسموح فقط لـ `Draft` و `UpdatedWithPrices`. بعد `Printed`، يُعرض العرض للقراءة فقط — يمنع التعديل لأن الطباعة تعني اعتماداً نهائياً للعرض.

### 3.2 تفصيل

| البند | القيمة |
|:------|:--------|
| **WF-ID** | `WF-01` |
| **الاسم** | تسجيل الدخول والإعداد الأولي |
| **المستخدم** | يزيد ماهر (Admin) |
| **متى يحدث؟** | كل تشغيل للتطبيق + أول مرة فقط (First Time Setup) |

#### الإجراءات

| الجهة | الإجراء | التفاصيل |
|:------|:--------|:---------|
| **النظام** | فحص وجود SQLite.db | عند تشغيل `App.xaml.cs` → `OnStartup` |
| **النظام** | إنشاء قاعدة البيانات | استخدام `EnsureCreated()` أو `Database.Migrate()` إذا لم تكن موجودة |
| **النظام** | التحقق من وجود مستخدم | استعلام `Users` — إذا فارغ → وضع FirstTimeSetup |
| **المستخدم** | إدخال كلمة المرور (الدخول العادي) | حقل `PasswordBox` في LoginView |
| **النظام** | التحقق من كلمة المرور | `AuthenticationService.LoginAsync("admin", password)` — مقارنة Hash |
| **النظام** | إنشاء أول مستخدم | `AuthenticationService.SetupFirstUserAsync(password)` — Username = "يزيد ماهر" |
| **النظام** | فتح MainWindow | إغلاق LoginView، عرض MainWindow مع Sidebar + Content Frame |
| **النظام** | التحقق من Auto Backup | بعد فتح MainWindow، تشغيل BackupService.CreateBackupAsync() في الخلفية |

#### قرارات العمل

| القرار | الخيارات | المسار |
|:-------|:---------|:--------|
| هل أول مرة تشغيل؟ | نعم / لا | نعم → FirstTimeSetup. لا → Login عادي |
| هل كلمة المرور صحيحة؟ | نعم / لا | نعم → فتح. لا → عرض خطأ + إعادة المحاولة (محاولات غير محدودة) |
| هل المستخدم يريد تغيير كلمة المرور؟ | من Settings | بعد تسجيل الدخول → شاشة الإعدادات → تغيير كلمة المرور |

#### حالات الخطأ

| الحالة | الإجراء |
|:-------|:--------|
| **فشل إنشاء SQLite.db** | عرض رسالة خطأ: "تعذر إنشاء قاعدة البيانات. تحقق من صلاحيات المجلد." ← إغلاق التطبيق |
| **فشل فتح اتصال SQLite** | عرض خطأ + تسجيل + محاولة إعادة الاتصال مرة واحدة |
| **خطأ غير متوقع أثناء Login** | تسجيل + رسالة "حدث خطأ غير متوقع" |
| **نسيت كلمة المرور؟** | لا خيار لاسترجاعها — الحل الوحيد: حذف ملف SQLite.db (فقدان البيانات) أو استرجاع Backup قديم |

## 14. Edge Cases وحالات الخطأ

### 14.1 ماذا لو فتح التطبيق ولم توجد قاعدة بيانات؟

| السيناريو | الإجراء |
|:---------|:--------|
| **أول تشغيل** | النظام ينشئ `SQLite.db` تلقائياً + ينشئ الجداول عبر `EnsureCreated()` |
| **تم حذف الملف يدوياً** | النظام يعتبره "أول تشغيل" — ينشئ قاعدة بيانات جديدة فارغة. المستخدم يفقد جميع البيانات. |
| **المسار غير صحيح** | (لا يحدث — المسار ثابت: `%LOCALAPPDATA%\TeraQuotation\Data\TeraQuotation.db`) |
| **الملف موجود لكن تالف** | استثناء عند فتح الاتصال ← رسالة "قاعدة البيانات تالفة — يرجى استرجاع نسخة احتياطية" ← فتح شاشة Restore |

### 14.2 ماذا لو تعذر فتح اتصال SQLite؟

| السيناريو | الإجراء |
|:---------|:--------|
| **الملف مقفل (Locked)** | SQLite يسمح بقراءة متزامنة لكن الكتابة متسلسلة. استثناء `SQLiteException` مع `SQLITE_BUSY` ← إعادة المحاولة بعد 500ms (حتى 3 مرات) |
| **الملف تالف (Corrupt)** | رسالة "قاعدة البيانات تالفة" ← عرض زر "استرجاع نسخة احتياطية" |
| **صلاحيات غير كافية** | رسالة "تعذر الوصول إلى قاعدة البيانات — تحقق من صلاحيات المجلد" ← إغلاق التطبيق |

### 14.4 ماذا لو تم حذف مورد/قطعة لها عروض سابقة؟

| السيناريو | item | المورد |
|:----------|:-----|:--------|
| **النوع** | FK مع `DeleteBehavior.Restrict` | نص — لا علاقة FK |
| **هل يمنع الحذف؟** | ✅ نعم — استثناء `DbUpdateException` | ❌ لا — الحذف آمن |
| **رسالة الخطأ** | "لا يمكن حذف هذه القطعة لأنها مستخدمة في [N] عرض سعر" | — (لا رسالة، يُحذف مباشرة) |
| **البديل** | Soft Delete (إضافة `IsDeleted` flag) — اختياري | — |
| **التأثير على العروض القديمة** | يظهر اسم القطعة في العروض القديمة (لكن `ItemId` ما زال موجوداً) | لا تأثير — الاسم نُسخ كنص وقت الإنشاء |

**ملاحظة مهمة:** حذف قطعة يمنع لأن `ItemId` FK. لكن الحذف قد يسبب مشكلة في عرض العروض القديمة (لن يتمكن `Include(qi.Item)` من جلب القطعة المحذوفة).
**توصية:** بدلاً من منع الحذف، يمكن استخدام Soft Delete (إضافة `bool IsDeleted` مع `[NotMapped]` Navigation Property يعرض null إذا محذوفة).

### 14.5 ماذا لو حاول المستخدم طباعة عرض بدون بيانات؟

| الحالة | الإجراء |
|:-------|:--------|
| **عرض جديد (لم يحفظ بعد)** | حفظ تلقائي أولاً ← طباعة |
| **عرض بدون قطع** | منع الطباعة: "يجب إضافة قطعة واحدة على الأقل" |
| **طباعة نهائية بدون أسعار** | تحذير: "بعض القطع ليس لها أسعار — هل تريد المتابعة؟" |
| **طباعة PDF بدون Status مناسب** | منع: "يجب أن يكون العرض محدّثاً بالأسعار أولاً" |

### 14.6 سيناريوهات إضافية

| السيناريو | الإجراء |
|:----------|:--------|
| **التطبيق يُغلق فجأة أثناء تحرير عرض** | التغييرات غير المحفوظة تُفقد — لأن الحفظ يدوي. يُعرض تحذير عند الإغلاق (`OnClosing` event) |
| **نفاد الذاكرة** | (غير متوقع لحجم المشروع) — في حالة حدوثه، `OutOfMemoryException` ← تسجيل + إعادة تشغيل |
| **مساحة القرص ممتلئة — فشل حفظ / Backup** | رسالة: "مساحة القرص غير كافية — حرر مساحة وحاول مرة أخرى" |

| **نظام التاريخ — هل يدعم التقويم الهجري؟** | لا — تقويم ميلادي فقط. العميل لم يطلب هجري. |

```
│ Outlook │                     │            │  (طباعة) │  (PDF)     │  (إعادة إرسال)│
└─────────┴─────────────────────┴────────────┴──────────┴────────────┴────────────────┘
```

### 15.2 قواعد الانتقال (Transition Rules)

| القاعدة | الشرح |
|:--------|:-------|
| **T1** | `Draft` → أي حالة أخرى: مسموح بعد إدخال الأسعار (عند الطباعة/PDF/Outlook يمر على `UpdatedWithPrices` أولاً) |
| **T2** | لا رجوع من `UpdatedWithPrices` إلى `Draft` — العرض اعتُبر مكتملاً |
| **T3** | لا رجوع من `Printed`, `PDFExported`, `SentViaOutlook` إلى حالات سابقة — الطباعة تعني اعتماداً نهائياً |
| **T4** | `Printed` → `PDFExported`: مسموح (تصدير PDF بعد الطباعة) |
| **T5** | `Printed` → `SentViaOutlook`: مسموح (إرسال بعد الطباعة) |
| **T6** | `PDFExported` → `Printed`: مسموح (طباعة بعد PDF) |
| **T7** | `Printed` → `Printed`: مسموح (إعادة طباعة) |
| **T8** | `PDFExported` → `PDFExported`: مسموح (إعادة تصدير) |
| **T9** | `SentViaOutlook` → `SentViaOutlook`: مسموح (إعادة إرسال) |
| **T10** | أي حالة ← `Draft`: **ممنوع** — التعديل بعد الاعتماد غير مسموح |

### 15.3 متى يتغيّر Status تلقائياً؟

| الإجراء | الانتقال | آلي أم يدوي |
|:--------|:----------|:-----------:|
| حفظ عرض جديد | → `Draft` | آلي |
| طباعة نهائية (لأول مرة) | `Draft`/`UpdatedWithPrices` → `Printed` | آلي (بعد نجاح الطباعة) |
| تصدير PDF | → `PDFExported` | آلي (بعد نجاح التصدير) |
| إرسال Outlook | → `SentViaOutlook` | آلي (بعد فتح نافذة Outlook) |
| طباعة بدون أسعار | **لا يتغير** | — |

## 16. قرارات العمل (Business Decision Points)

### 16.1 قائمة القرارات المؤرشفة

| # | القرار | التصنيف | الأساس | ملاحظات |
|:-:|:-------|:-------:|:-------|:--------|
| 1 | **Supplier في QuotationItem نصوص (وليست FK)** | معماري | حماية البيانات التاريخية — إذا غيّر المورد اسمه، تبقى العروض القديمة كما هي | تم اعتماده في ADR-005 |
| 2 | **حذف قطعة يمنع إذا مستخدمة في عروض** | عمل | الحفاظ على سلامة البيانات (FK Restrict) | التوصية: استخدام Soft Delete بدلاً من المنع المطلق |
| 3 | **التعديل بعد الطباعة ممنوع** | عمل | الطباعة تعني اعتماداً نهائياً | يمكن فتح العرض للقراءة فقط |
| 4 | **Outlook يفتح نافذة — لا إرسال تلقائي** | أمان | المستخدم يراجع البريد قبل الإرسال | Outlook Interop يفتح `MailItem.Display()` — ليس `Send()` |
| 5 | **PDF يفتح SaveFileDialog — لا مسار افتراضي** | سهولة استخدام | المستخدم يختار مكان الحفظ | الاسم الافتراضي: `Q-001_2026-07-13.pdf` |
| 6 | **Auto Backup عند فتح التطبيق فقط** | أداء | لا حاجة لـ Backup مؤقت — البيانات تتغير فقط عند الحفظ | العمق: يحتفظ بآخر 10 نسخ |
| 7 | **لا Pagination معقدة — Only Skip/Take** | أداء | كمية البيانات صغيرة (< 10,000 عرض) | تكفي 20 عرضاً لكل صفحة |
| 8 | **تسلسل رقمي: Q-001, Q-002, ...** | عمل | بسيط، مقروء، يتجاوز Q-999 تلقائياً | لا إعادة تعيين في بداية كل سنة |
| 9 | **التقارير تستثني المسودات (Draft)** | عمل | المسودات غير مكتملة — لا تدخل في الإحصائيات | D3: اختياري (يمكن تضمينها) — D4: إلزامي الاستثناء |

### 16.2 نقاط قرار مستقبلية (تحتاج موافقة)

| # | القرار المعلق | السيناريو | التأثير |
|:-:|:--------------|:----------|:--------|
| 1 | **Soft Delete للقطع** | إذا أراد المستخدم حذف قطعة مستخدمة في عروض سابقة | بدون Soft Delete: يمنع الحذف. مع Soft Delete: يسمح لكن القطعة تختفي من ComboBox |
| 2 | **فتح عرض بعد الطباعة للتعديل** | إذا احتاج المستخدم تعديل عرض مطبوع | حالياً: ممنوع. قد نحتاج إضافة "نسخ كعرض جديد" |
| 3 | **إعادة تعيين الرقم التسلسلي سنوياً** | مثلاً: Q-2026-001, Q-2027-001 | حالياً: Q-001 مستمر. التغيير يؤثر على التنسيق |

## 17. Gaps and Recommendations

### 17.1 Design Gaps

| # | الفجوة | النوع | التأثير | التوصية |
|:-:|:-------|:-----:|:--------|:--------|
| 1 | **لا يوجد `12_BUSINESS_RULES.md`** | 📄 مفقود | بعض القواعد التجارية (مثل: هل يسمح بسعر 0؟ هل يمكن تعديل عرض مطبوع؟) موثقة هنا فقط | يُنشأ ملف Business Rules لتوثيق القرارات رسمياً |
