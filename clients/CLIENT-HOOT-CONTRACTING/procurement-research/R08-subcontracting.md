## مقدمة

**Subcontracting** (التعهيد / مقاولي الباطن) هو عملية يقوم بموجبها المقاول الرئيسي (Prime Contractor / General Contractor) بإسناد جزء من أعمال المشروع إلى مقاول آخر متخصص (Subcontractor) مع الاحتفاظ بالمسؤولية التعاقدية الكاملة تجاه المالك (Client/Owner).

في أنظمة ERP، تختلف معالجة المشتريات من مقاولي الباطن بشكل جوهري عن المشتريات التقليدية (مواد / خدمات مباشرة) لأنها تتطلب:
- توفير المواد الخام للمقاول (Material Provisioning)
- تتبع استهلاك المواد (Component Consumption)
- إدارة جداول الدفع المرتبطة بالإنجاز (Progress Payments)
- ربط العقود مع كشوف الكميات (Bill of Quantities - BOQ)
- إدارة الاحتجازات الضمانية (Retentions / Holdbacks)

## SAP MM Subcontracting

### 1.1 دورة Subcontracting الكاملة (Subcontracting Cycle)

```
┌─────────────────────────────────────────────────────────────┐
│                SAP MM Subcontracting Cycle                   │
├─────────────────────────────────────────────────────────────┤
│  الخطوة 1: إنشاء أمر شراء تعهيد (Subcontracting PO)         │
│    ME21N - Item Category = L (Subcontracting)               │
├─────────────────────────────────────────────────────────────┤
│  الخطوة 2: توفير المواد الخام للمقاول (Provision of        │
│    Components)                                              │
│    MB1B - Movement Type 541 (Provision to Vendor)           │
├─────────────────────────────────────────────────────────────┤
│  الخطوة 3: استلام المنتج النهائي (Goods Receipt)            │
│    MIGO - Movement Type 101                                 │
│    ⟶ يستهلك المكونات تلقائيًا (Automatic Reversal 542)     │
├─────────────────────────────────────────────────────────────┤
│  الخطوة 4: إصدار الفاتورة (Invoice Receipt)                │
│    MIRO - FI Settlement                                     │
├─────────────────────────────────────────────────────────────┤
│  الخطوة 5: تسوية المخزون مع المقاول (Stock Reconciliation) │
│    ME2O / ME2ON - Subcontracting Cockpit                   │
└─────────────────────────────────────────────────────────────┘
```

#### تفصيل الدورة كاملة:

**أ. إنشاء أمر شراء تعهيد (Subcontracting Purchase Order)**
- TCode: `ME21N` (Create), `ME22N` (Change), `ME23N` (Display)
- Item Category: **L** (Subcontracting) — هذا هو العنصر الأساسي الذي يفرق بين أمر الشراء العادي وأمر شراء التعهيد
- يتم تحديد المادة النهائية (Finished Product) التي سيقوم المقاول بتصنيعها
- يتم تحديد كمية الأمر (Order Quantity)
- يتم ربط **Bill of Materials (BOM)** الخاص بالمادة النهائية تلقائيًا لتحديد المكونات التي سيتم توفيرها للمقاول

**ب. توفير المواد الخام للمقاول (Material Provisioning to Subcontractor)**
- TCode: `MB1B` مع Movement Type **541** (Issue to Vendor / Provision to Subcontractor)
- يتم ترحيل المواد الخام من مخزون الشركة إلى مخزون المقاول (Stock of Material Provided to Vendor)
- يمكن التتبع عبر Stock Type: **Vendor Consignment / Subcontracting Stock**
- يتم خصم المواد من المخزون المتاح (Unrestricted Use Stock)

```
حركة 541:
  المخزون المتاح (Unrestricted)         مخزون التعهيد (Subcontracting Stock)
  [مواد خام] ──────────────────────────► [مواد خام عند المقاول]
```

**ج. استهلاك المكونات والترحيل إلى المخزون (Goods Receipt & Consumption)**
- TCode: `MIGO` (Goods Receipt for Purchase Order) مع Movement Type **101**
- عند استلام المنتج النهائي من المقاول، يقوم SAP تلقائيًا بـ:
  1. ترحيل المنتج النهائي إلى المخزون (GR)
  2. استهلاك المكونات التي تم توفيرها سابقًا للمقاول (Automatic Reversal / Consumption)
  3. الترحيل يتم كحركة **542** عكسية لـ 541 تلقائيًا

```
حركة 101 (GR):
  المخزون [منتج نهائي] ◄── تم الاستلام من المقاول
  
حركة 542 (تلقائية):
  مخزون التعهيد [مواد خام] ◄── استهلاك (Consumption)
  يتم ترحيل القيد المحاسبي:
    Dr. Inventory (Finished Product)
    Cr. GR/IR Clearing
    Dr. Consumption Account (Components)
    Cr. Subcontracting Stock
```

**د. إصدار الفاتورة والترحيل المحاسبي مع FI (Invoice Receipt & FI Settlement)**
- TCode: `MIRO` (Enter Invoice for Purchase Order)
- يتم إصدار فاتورة من المقاول على قيمة خدمات التصنيع (Service/Conversion Fee) وليس قيمة المواد الخام (لأن المواد مملوكة للشركة الأم)
- الترحيل المحاسبي:

```
Dr. GR/IR Clearing
    Cr. Vendor (Accounts Payable)
    
(يتم تسوية GR/IR لاحقًا عند مطابقة الفاتورة مع أمر الشراء والإذن الاستلام)
```

### 1.2 أكواد المعاملات الرئيسية (Key TCodes)

| TCode | الوظيفة | الشرح |
|-------|---------|-------|
| **ME21N** | Create Purchase Order | إنشاء أمر شراء (بند من نوع L للتعهيد) |
| **ME22N** | Change Purchase Order | تعديل أمر شراء تعهيد |
| **ME2O** | Stock Provided to Vendor | تقرير المخزون المقدم للمقاولين |
| **ME2ON** | Subcontracting Cockpit | لوحة متابعة التعهيد — تعرض المواد المقدمة لكل مقاول وأمر شراء |
| **MB1B** | Transfer Posting | ترحيل نقل المخزون (حركات 541/542) |
| **MIGO** | Goods Movement | استلام البضائع وإصدار المواد |
| **MIRO** | Invoice Receipt | إصدار الفاتورة والترحيل المحاسبي |
| **CS01** | Create BOM (Bill of Materials) | إنشاء شجرة المكونات للمادة النهائية |
| **CS02** | Change BOM | تعديل شجرة المكونات |
| **ME2L** | Purchasing Documents by Vendor | أوامر الشراء حسب المورد |
| **MB51** | Material Document List | كشف حركات المواد |
| **FBL1N** | Vendor Line Items | بنود المورد المحاسبية |

### 1.3 الجوانب المحاسبية (Accounting & Settlement)

**حسابات التعهيد في SAP (Automatic Account Determination):**
- **BSX (Inventory Posting):** حساب المخزون للمنتج النهائي
- **GBB-BSA (Consumption):** حساب استهلاك المكونات
- **WRX (GR/IR Clearing):** حساب تسوية الاستلام والفواتير
- **KDM (Material Provided to Vendor):** حساب المواد المقدمة للمقاول

**التأثير على التكاليف (Costing Impact):**
- تكلفة المنتج النهائي = تكلفة المكونات المستهلكة + رسوم التصنيع (Conversion Cost / Service Charges)
- في SAP، يتم حساب التكلفة عبر Material Ledger / Actual Costing

## Oracle Procurement Subcontracting

### 2.1 نظرة عامة على Subcontracting في Oracle

في **Oracle Fusion Cloud Procurement** و **Oracle Project Portfolio Management (PPM)**، تتم إدارة مشتريات مقاولي الباطن عبر:

1. **Subcontracting Agreements** — اتفاقيات إطارية مع مقاولي الباطن
2. **Subcontracting Purchase Orders** — أوامر شراء خدمات الباطن
3. **Project Contracts** — عقود المشاريع المرتبطة بالمقاولين
4. **Change Orders** — أوامر التعديل على نطاق العمل أو القيمة
5. **Progress Payments** — الدفعات المرحلية المرتبطة بالإنجاز

### 2.2 إدارة عقود الباطن (Subcontract Management)

Oracle يدير عقود الباطن عبر وحدة **Procurement Contracts**:

**أنواع العقود في Oracle:**
| نوع العقد | الوصف |
|-----------|-------|
| **Contract Purchase Agreement** | اتفاقية شراء إطارية لفترة زمنية مع تسعير محدد |
| **Blanket Purchase Agreement (BPA)** | اتفاقية شراء مظلة بكميات/قيم تقديرية |
| **Subcontract Agreement** | عقد مقاول باطن محدد بنطاق عمل ومدة |
| **Project Contract** | عقد مشروع مع جدولة دفعات وإنجاز |

**مكونات عقد الباطن في Oracle:**
- **Contract Header**: معلومات العقد الأساسية (المقاول، المشروع، المدة، القيمة)
- **Contract Lines**: بنود العقد — كل بند يمثل حزمة عمل (Work Package) أو بند كشف كميات (BOQ Line)
- **Terms & Conditions**: الشروط والأحكام العامة والخاصة
- **Payment Terms**: شروط الدفع — دفعات مرحلية، دفعة أولى، حجز ضمان
- **Milestones**: المراحل الرئيسية المرتبطة بالدفعات
- **Change Orders**: أوامر التعديل — زيادة/نقصان في النطاق أو القيمة

### 2.3 التكامل مع Project Costing

Oracle يوفر تكاملًا عميقًا مع **Project Costing** عبر:

```
Subcontract PO ──► Project Expenditure ──► Project Costing
                      │
                      ├── Labor (تكاليف العمالة)
                      ├── Materials (تكاليف المواد)
                      ├── Equipment (تكاليف المعدات)
                      └── Subcontract Services (خدمات الباطن)
```

**كيفية التكامل:**
1. يتم إنشاء أمر شراء الباطن وربطه برقم المشروع (Project Number) ورقم المهمة (Task Number)
2. عند استلام الفاتورة، يتم تحميل التكلفة تلقائيًا على المشروع
3. يتم توزيع التكاليف على بنود العمل (Work Breakdown Structure - WBS)
4. يمكن تتبع التكاليف التقديرية مقابل الفعلية (Budget vs Actual) عبر Project Costing

### 2.4 Change Orders (أوامر التعديل)

Oracle يدير تغييرات عقود الباطن من خلال **Change Order Management**:

```
عقد باطن أصلي (Original Contract)
    │
    ├── Change Order #1: إضافة أعمال كهربائية إضافية (+ 50,000 ر.س)
    ├── Change Order #2: حذف أعمال تكييف غير ضرورية (- 20,000 ر.س)
    ├── Change Order #3: تعديل مواصفات العزل (+ 15,000 ر.س)
    └── Change Order #4: تمديد فترة العقد (بدون تكلفة)
```

**أنواع Change Orders:**
| النوع | الوصف |
|-------|-------|
| **Scope Change** | تغيير في نطاق العمل (إضافة / حذف) |
| **Price Change** | تغيير في السعر (تعديل أسعار الوحدات / الخصم) |
| **Schedule Change** | تغيير في الجدول الزمني (تمديد / تقصير) |
| **Contract Term Change** | تغيير في شروط العقد |

**سير عمل Change Order في Oracle:**
1. إنشاء طلب تغيير (Change Request)
2. تقييم التكلفة والأثر (Cost Impact Assessment)
3. اعتماد التغيير (Change Order Approval Workflow)
4. تعديل العقد الأصلي (Contract Amendment)
5. تحديث BOQ و Justifications

### 2.5 تتبع الدفعات والتسويات (Progress Payments & Settlement)

Oracle يدعم عدة آليات للدفع لمقاولي الباطن:

| آلية الدفع | الوصف |
|-----------|-------|
| **Progress Billing** | إصدار فواتير بناءً على نسبة الإنجاز المعتمدة |
| **Milestone Billing** | إصدار فواتير عند تحقيق مراحل محددة |
| **Time & Material Billing** | إصدار فواتير على أساس ساعات العمل والمواد الفعلية |
| **Completion Billing** | إصدار فاتورة عند اكتمال العقد بالكامل |
| **Advance Payment** | دفعة مقدمة (عادةً 5-15% من قيمة العقد) |

## Microsoft Dynamics 365 Project Operations

### 3.1 Project Operations وإدارة عقود الباطن

**Microsoft Dynamics 365 Project Operations** هو الحل الأمثل لشركات المقاولات والمشاريع. يوفر إدارة متكاملة لعقود الباطن عبر الوحدات التالية:

| الوحدة | الوظيفة في إدارة الباطن |
|--------|------------------------|
| **Project Contracts** | إدارة عقود الباطن — النطاق، القيمة، الشروط |
| **Project Procurement** | مشتريات المشروع — أوامر شراء خدمات الباطن |

| **Project Invoice** | فوترة المشاريع — إصدار فواتير لمالك المشروع وعقود الباطن |
| **Vendor Collaboration** | تعاون مع الموردين — بوابة لمقاولي الباطن |

### 3.3 Subcontractor Management في Supply Chain

في Dynamics 365 Supply Chain Management، يتم إدارة مقاولي الباطن كـ **Vendors** مع خصائص إضافية:

**خصائص مورد الباطن (Subcontractor Vendor Setup):**
```
مقاول الباطن:
  ├── Vendor Type: Subcontractor
  ├── Vendor Account Group: SUBCON
  ├── Procurement Class: Services / Construction
  ├── Currency: ريال سعودي / دولار
  ├── Payment Terms: Net 30 / Milestone Based
  ├── Withholding Tax: نعم (إذا كان مطلوبًا)
  └── Vendor Hold: لا (إذا كان معتمدًا)
```

**التحقق من تأهيل المقاول (Vendor Approval):**
- يمكن ربط مقاول الباطن بعملية **Vendor Prequalification**
- يتطلب إرفاق الشهادات والتراخيص (License, Insurance, Classification)
- صلاحية التصنيف تحد تلقائيًا (مثل تصنيف المقاولين في الهيئة السعودية)

## ما يميز مشتريات مقاولي الباطن في المقاولات

### 4.1 أنواع عقود الباطن في قطاع المقاولات

في المشاريع الإنشائية، تتنوع عقود الباطن حسب التخصص:

| التخصص | نطاق العمل النموذجي |
|--------|-------------------|
| **كهرباء (Electrical)** | تمديدات الكهرباء، اللوحات الرئيسية والفرعية، الإنارة، أنظمة الحماية من الصواعق |
| **تكييف (HVAC)** | أنظمة التكييف المركزي، مجاري الهواء، التبريد، التهوية |
| **دهانات (Painting)** | الدهانات الداخلية والخارجية، العوازل الحرارية |
| **عزل (Waterproofing)** | عزل الأسطح، عزل الخزانات، عزل الحمامات، العزل الصوتي |
| **سباكة (Plumbing)** | تمديدات المياه والصرف الصحي، الخزانات، المضخات |
| **أعمال إنشائية (Civil/Structural)** | الخرسانة، الحديد، القصارة، البلاط |
| **واجهات (Facade/Cladding)** | واجهات الزجاج والألمنيوم، الحجر |
| **أعمال حديدية (Metal Works)** | الهياكل المعدنية، الدرابزين، البوابات |
| **مكافحة حريق (Fire Fighting)** | أنظمة الإطفاء، الإنذار، المرشات |
| **مصاعد (Elevators)** | تركيب وصيانة المصاعد والسلالم المتحركة |

### 4.2 إدارة الإشراف والجودة لأعمال الباطن (Quality & Supervision)

إدارة جودة أعمال الباطن تشمل:

```
سير عمل الجودة:
  ┌─────────────────┐
  │ فحص المواد الواردة │ ◄── تأكد من مطابقة المواد للمواصفات
  └────────┬────────┘
           ▼
  ┌──────────────────────┐
  │ فحص أثناء التنفيذ     │ ◄── تفتيش دوري من مشرف الإشراف
  └────────┬────────┘
           ▼
  ┌──────────────────────────┐
  │ اختبارات ما بعد التنفيذ   │ ◄── اختبارات التشغيل والأداء
  └────────┬────────┘
           ▼
  ┌──────────────────────────────┐
  │ إصدار شهادة الإنجاز الجزئي/النهائي │
  └──────────────────────────────┘
```

**نقاط التفتيش (Inspection Hold Points):**
- يجب تحديد نقاط التفتيش الإجبارية في العقد
- لا يمكن لمقاول الباطن تجاوز هذه النقاط دون موافقة خطية
- أمثلة: صب الخرسانة، اختبار ضغط المواسير، اختبار تشغيل أنظمة التكييف

### 4.3 شهادات الإنجاز (Completion Certificates)

**أنواع شهادات الإنجاز:**

| نوع الشهادة | التوقيت | المعنى |
|------------|---------|--------|
| **Provisional/Practical Completion** | اكتمال الأعمال مع وجود هامش بسيط | يمكن تسليم الموقع للمالك مع أعمال تكميلية محدودة |
| **Sectional Completion** | اكتمال جزء / مرحلة | تسليم جزئي — مثل اكتمال الطابق الأرضي |
| **Final Completion** | اكتمال كامل واجتياز جميع الاختبارات | إنهاء العقد بالكامل مع إصدار شهادة الصيانة |
| **Defects Liability Certificate** | انتهاء فترة الصيانة | إخلاء مسؤولية مقاول الباطن من العيوب |
| **Taking-Over Certificate** | الاستلام الابتدائي | نقل الحيازة من المقاول لمالك المشروع |

**محتوى شهادة الإنجاز النموذجية:**
- اسم المشروع ورقم العقد
- اسم مقاول الباطن
- وصف الأعمال المنجزة
- تاريخ الإنجاز
- تاريخ بدء فترة الصيانة (Defects Liability Period)
- قيمة الأعمال المنجزة
- الاحتجاز الضماني المتبقي
- توقيع مدير المشروع ومقاول الباطن

### 4.4 Progress Payments لمقاولي الباطن

آلية الدفع التقدمي هي الأساس في عقود المقاولات:

```
┌────────────────────────────────────────────────────────────────────────┐
│                        Progress Payment Process                        │
├────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  1. إعداد تقرير الإنجاز (Progress Report)                              │
│     - نسبة الإنجاز الفعلية مقابل المخطط لها                            │
│     - كشف الكميات المنجزة (Measured Quantities)                        │
│                                                                         │
│  2. إعداد شهادة الدفع (Payment Certificate / Application for Payment)  │
│     - قيمة الأعمال المنجزة هذا الشهر (Work Done to Date)               │
│     - قيمة المواد في الموقع (Materials on Site)                        │
│     - خصم الدفعات السابقة (Less Previous Certificates)                 │
│     - خصم الاحتجاز الضماني (Less Retention % من إجمالي الإنجاز)        │
│                                                                         │
│  3. مراجعة واعتماد (Review & Approval)                                 │
│     - مراجعة من استشاري المشروع / مدير المشروع                         │
│     - التأكد من مطابقة الإنجاز للواقع                                  │
│                                                                         │
│  4. إصدار فاتورة مقاول الباطن (Subcontractor Invoice)                  │
│     - إصدار فاتورة بناءً على شهادة الدفع المعتمدة                      │
│                                                                         │
│  5. الصرف (Payment)                                                     │
│     - صرف الدفعة بعد خصم الاحتجاز                                       │
│     - إصدار سند صرف / حوالة بنكية                                      │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘
```

**مثال لجدول الدفع لمقاول كهرباء (قيمة العقد: 500,000 ر.س):**

| الشهر | الإنجاز % | قيمة الإنجاز | الدفعات المتراكمة | الاحتجاز 10% | صافي المستحق |
|-------|----------|-------------|------------------|-------------|-------------|
| 1 | 15% | 75,000 | 75,000 | 7,500 | 67,500 |
| 2 | 25% | 125,000 | 200,000 | 12,500 | 112,500 |
| 3 | 30% | 150,000 | 350,000 | 15,000 | 135,000 |
| 4 | 20% | 100,000 | 450,000 | 10,000 | 90,000 |
| 5 | 10% | 50,000 | 500,000 | 5,000 | 45,000 |
| الصيانة (بعد 12 شهر) | — | إفراج الاحتجاز | 500,000 | 50,000 | 50,000 |
| **الإجمالي** | **100%** | **500,000** | | | **500,000** |

### 4.5 Retentions (حجز ضمان 5-10%)

**Retention / Holdback** هو مبلغ يتم اقتطاعه من كل دفعة لمقاول الباطن ويتم الاحتفاظ به كضمان لضمان إتمام الأعمال بشكل صحيح.

**خصائص الاحتجاز الضماني:**
| الخاصية | القيمة النموذجية |
|---------|-----------------|
| نسبة الاحتجاز | 5% - 10% من قيمة شهادة الدفع |
| الحد الأقصى | 5% - 10% من إجمالي قيمة العقد |
| فترة الاحتجاز | حتى اكتمال الأعمال + فترة الصيانة (عادة 12 شهرًا) |
| تاريخ الإفراج | بعد إصدار شهادة الإنجاز النهائي + فترة الصيانة |

**أنواع الاحتجاز في ERP:**
```
System Type A: احتجاز على مستوى فاتورة المقاول
  - يتم اقتطاع النسبة من كل فاتورة
  - التتبع يتم على مستوى Vendor Invoice

System Type B: احتجاز على مستوى العقد
  - يتم ربط الاحتجاز ببنود العقد
  - التتبع يتم على مستوى Contract Line

System Type C: احتجاز مؤقت على مستوى المشروع
  - الاحتجاز على مستوى Project
  - لكل مقاول باطن احتجاز محدد بحسب عقده
```

**المعالجة المحاسبية للاحتجاز:**
```
عند إصدار فاتورة الباطن:
  Dr. Project Cost (قيمة العقد)
  Cr. Vendor Account (صافي الدفعة بعد خصم الاحتجاز)
  Cr. Retention Payable (مبلغ الاحتجاز)

عند الإفراج عن الاحتجاز:
  Dr. Retention Payable (مبلغ الاحتجاز)
  Cr. Bank / Cash (صرف الاحتجاز لمقاول الباطن)
```

### 4.6 Scope of Work (SOW) مقابل Bill of Quantities (BOQ)

**Scope of Work (SOW)** — نطاق العمل:
- وصف نصي ووظيفي للأعمال المطلوبة
- يحدد المسؤوليات والتسليمات (Deliverables)
- غير قابل للقياس الكمي المباشر

**Bill of Quantities (BOQ)** — كشف الكميات:
- قائمة رقمية مفصلة ببنود الأعمال
- كل بند له: رقم، وصف، وحدة قياس، كمية، سعر الوحدة، السعر الإجمالي
- يستخدم كأساس للعطاءات (Tendering) والدفعات (Progress Payments)

**العلاقة بين SOW و BOQ في أنظمة ERP:**

```
SOW (Scope of Work)
  │
  ├── بنود BOQ التابعة:
  │     ├── BOQ-001: تمديدات الكهرباء — 1,000 م2
  │     ├── BOQ-002: لوحات توزيع — 15 وحدة
  │     ├── BOQ-003: نقاط إضاءة — 200 نقطة
  │     └── BOQ-004: اختبارات التشغيل — 1 lump sum
  │
  └── مدة العقد: 6 أشهر
```

**في SAP:** يتم تمثيل BOQ عبر Outline Agreement مع Item Categories محددة.

**في Oracle:** BOQ يمثل Contract Lines مع Unit Price و Quantity.

**في Dynamics 365:** BOQ يمثل Project Contract Lines مع Billing Method = Fixed Price / T&M.

### 5.3 جداول المشتريات حسب الجدول الزمني للمشروع (Critical Path)

**ربط المشتريات بجدول المشروع:**

```
Critical Path ──► أنشطة المسار الحرج
                      │
                      ├── نشاط A: صب الخرسانة (أسابيع 1-4)
                      │     └── شراء الخرسانة الجاهزة ← مقاول الخرسانة
                      │
                      ├── نشاط B: تمديدات الكهرباء (أسابيع 5-10)
                      │     └── شراء خدمات الكهرباء ← مقاول الكهرباء
                      │     └── شراء الكابلات والمفاتيح ← مورد توريد
                      │
                      ├── نشاط C: أعمال التكييف (أسابيع 6-12)
                      │     └── شراء خدمات التكييف ← مقاول التكييف
                      │
                      └── نشاط D: التشطيب (أسابيع 11-16)
                            └── شراء خدمات الدهانات ← مقاول الدهانات
                            └── شراء خدمات السباكة ← مقاول السباكة
```

**مصفوفة أولويات المشتريات حسب المسار الحرج:**

| حالة النشاط | أولوية المشتريات | الإجراء |
|------------|-----------------|---------|
| **On Critical Path** | **حرجة — يجب البدء فورًا** | متابعة يومية، التنسيق المسبق مع المقاول |
| Near Critical (Float < 10%) | مرتفعة | متابعة أسبوعية |
| Non-Critical (Float > 20%) | عادية | متابعة شهرية |

**تكامل جدول المشتريات مع ERP:**
- يتم تحميل جدول المشروع (Project Schedule) في النظام
- يتم ربط كل بند شراء (Purchase Order) بنشاط محدد (Activity / Task)
- عند تأخير نشاط حرج، يتم إشعار فريق المشتريات تلقائيًا
- يمكن استخدام **Earned Value Management (EVM)** لربط الإنجاز الفعلي بالمدفوعات

### 5.4 NCR (Non-Conformance Reports)

**NCR** — تقرير عدم المطابقة هو وثيقة رسمية تُصدر عند اكتشاف عمل غير مطابق للمواصفات.

**أنواع NCR:**
| النوع | الوصف |
|-------|-------|
| **Minor NCR** | عدم مطابقة بسيط — قابل للتصحيح دون تأثير على الجدول أو الجودة |
| **Major NCR** | عدم مطابقة رئيسي — يتطلب تصحيحًا ويؤثر على الجدول أو التكلفة |
| **Critical NCR** | عدم مطابقة خطير — يؤثر على سلامة المنشأة أو السلامة العامة |

**سير عمل NCR في نظام ERP:**

```
┌─────────────────────────────────────────────────────────────────┐
│                      NCR Process Flow                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  1. اكتشاف عدم المطابقة (Non-Conformance Identified)            │
│     - يتم إصدار NCR من مشرف الجودة / الاستشاري                   │
│     - توثيق: صور، قياسات، تقارير اختبار                         │
│                                                                   │
│  2. فتح NCR في النظام (NCR Creation)                            │
│     - رقم NCR تلقائي — تاريخ — موقع — وصف — درجة الخطورة       │
│     - ربط NCR مع مقاول الباطن ورقم العقد ورقم الأمر             │
│     - رفع المستندات الداعمة                                      │
│                                                                   │
│  3. تحليل السبب الجذري (Root Cause Analysis)                     │
│     - مقاول الباطن يقدم تحليل السبب                              │
│     - خطة العمل التصحيحي (Corrective Action Plan - CAP)         │
│                                                                   │
│  4. اعتماد خطة التصحيح (CAP Approval)                           │
│     - مراجعة من مدير المشروع / استشاري                          │
│     - الموافقة على خطة التصحيح                                   │
│                                                                   │
│  5. تنفيذ التصحيح (Corrective Action Execution)                  │
│     - مقاول الباطن ينفذ التصحيح                                  │
│     - إعادة الفحص والاختبار                                      │
│                                                                   │
│  6. إغلاق NCR (NCR Closure)                                     │
│     - التحقق من فعالية التصحيح                                   │
│     - إصدار تقرير إغلاق NCR                                      │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**حقول NCR في نظام ERP:**

| الحقل | الوصف |
|-------|-------|
| NCR Number | رقم تقرير عدم المطابقة (تلقائي) |
| Project | المشروع المرتبط |
| Subcontractor | مقاول الباطن المخالف |
| Contract PO | أمر الشراء / العقد المرتبط |
| Location | موقع عدم المطابقة |
| Description | وصف عدم المطابقة |
| Severity | درجة الخطورة — Minor / Major / Critical |
| Root Cause | السبب الجذري |
| Corrective Action | الإجراء التصحيحي |
| Responsible Party | الطرف المسؤول عن التصحيح |
| Target Date | تاريخ الإغلاق المخطط |
| Actual Closure Date | تاريخ الإغلاق الفعلي |
| Status | الحالة — مفتوح / تحت التصحيح / مغلق |
| Attachments | المستندات الداعمة (صور، تقارير) |

**ارتباط NCR مع الدفعات:**
- يمكن ربط NCR مع عملية الدفع — لا يتم صرف أي دفعة لمقاول الباطن قبل إغلاق NCRs المفتوحة
- يتم خصم تكاليف إعادة العمل (Rework Cost) من دفعات مقاول الباطن

## الخلاصة والتوصيات

### النتائج الرئيسية:

1. **Subcontracting** يختلف جوهريًا عن المشتريات التقليدية في ERP ويتطلب معالجة خاصة للمخزون المقدم للمقاولين، والدفعات التقدمية، والاحتجازات الضمانية.

2. **SAP MM** يوفر أقوى معالجة للمواد والمخزون في التعهيد عبر Item Category L و Material Provisioning، لكنه أضعف في إدارة العقود والدفعات التقدمية بدون تخصيص إضافي.

3. **Oracle Procurement** يوفر تكاملًا ممتازًا مع Project Costing و Change Orders، وهو الأنسب للمؤسسات التي تستخدم Oracle PPM بالفعل.

4. **Dynamics 365 Project Operations** يوفر الحل الأكثر تكاملًا لشركات المقاولات بميزات قوية مثل Time & Expense و Progress Billing و Subcontractor Portals مع سهولة الاستخدام.

5. **أفضل الممارسات** تشمل: تأهيل المقاولين عبر نظام موثق، ربط المشتريات بالمسار الحرج، إدارة NCRs بشكل منهجي، وفصل الاحتجازات الضمانية محاسبيًا.
