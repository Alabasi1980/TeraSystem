## 1. المبدأ التشغيلي

مساحة عمل التطبيق تعمل مثل **شركة مقاولات**:

- **المالك**: Majed، خارج الجلسات، يقرر يدويًا متى تعمل كل جلسة ويعتمد أو يرفض.
- **المقاول الرئيسي**: `tera`، يقود التنفيذ ويدير العملاء الفرعيين داخل جلسة القيادة.
- **الإشراف/الجودة**: `auditor`، يراجع جودة العمل وما تم توثيقه، ويعمل commit محلي فقط بعد قبول المالك الصريح.
- **المراقبة العامة**: `monitor`، يراجع توافق العمل مع الخطة الرئيسية والتفصيلية واكتشاف الانحرافات.
- **مراجعة التصميم**: `design-reviewer`، يراجع الالتزام البصري والتصميمي عند وجود مهام UI/UX.

لا تعمل أي جلسة رقابية تلقائيًا. المالك يقرر يدويًا متى يفتح أو يطلب عمل كل جلسة.

## 3. الجلسات الرئيسية

| الجلسة | المسؤول | النوع | الوظيفة |
|---|---|---|---|
| `tera` | Tera Agent | قيادة | تجهيز المرحلة، تفويض العملاء الفرعيين، التوثيق، انتظار موافقة المالك |
| `auditor` | Auditor Agent | جودة + Git محلي | تدقيق النتيجة، كشف الخلل، توثيق الملاحظات، commit بعد قبول المالك |
| `monitor` | Monitor Agent | رقابة خطة | مراجعة التوافق مع `PROJECT_MASTER_PLAN.md` و`PROJECT_DETAILED_EXECUTION_PLAN.md` |
| `design-reviewer` | Design Reviewer Agent | تدقيق بصري | مراجعة التصميم والتجربة البصرية عند وجود UI/UX |

## 4. حدود المسؤولية

### 4.1 Tera

- يقود التنفيذ ولا يعتمد نفسه نهائيًا دون بوابات المراجعة المطلوبة.
- يدير العملاء الفرعيين التنفيذيين داخل جلسة القيادة.
- يوثق نتائج العمل في `project-control/`.
- ينتظر موافقة المالك قبل الانتقال للمرحلة أو المهمة التالية.

### 4.2 Auditor

- لا ينفذ مزايا جديدة.
- لا يغير الخطة أو النطاق.
- يراجع الملفات المتغيرة والسجلات ذات العلاقة.
- يكتب ملاحظاته كتقرير للمالك أو في ملف مفوض له.
- يعمل commit محلي فقط بعد قبول المالك الصريح للمرحلة أو المهمة.
- لا يعمل push أبدًا دون موافقة صريحة منفصلة.

### 4.3 Monitor

- لا يراجع جودة الكود تفصيليًا.
- يراجع هل العمل مطابق للخطة الرئيسية والتفصيلية.
- يكشف: مهمة ناقصة، انحراف نطاق، ترتيب خاطئ، تجاوز موافقات، أو خطة غير مكتملة.
- يرفع تقريره للمالك ولا يصحح التنفيذ مباشرة.

### 4.4 Design Reviewer

- لا يصمم ولا ينفذ UI.
- يراجع الالتزام بالتصميم المعتمد فقط.
- يستخدم المعاينة البصرية أو fetch أو تشغيل التطبيق عند الحاجة وبموافقة المالك.
- لا يعتمد أي مهمة تصميمية إلا كتقرير مراجعة للمالك.

---

## 5. دورة العمل المعتمدة

1. Tera يجهز المهمة أو المرحلة.
2. Tera ينفذ أو يفوض عميلًا فرعيًا.
3. Tera يوثق النتيجة في `project-control/`.
4. المالك يقرر يدويًا إن كان سيطلب مراجعة من `auditor` أو `monitor` أو `design-reviewer`.
5. الجلسة الرقابية تراجع وتكتب تقريرًا للمالك.
6. إذا وُجد خلل، يعود المالك إلى Tera بطلب تصحيح.
7. إذا قبل المالك المرحلة، يمكن لـ `auditor` تنفيذ commit محلي عند طلب المالك الصريح.
8. لا ينتقل Tera للمرحلة التالية إلا بعد موافقة المالك.

---

## 6. قواعد منع التضخم

- لا ملف جديد إلا إذا لم يكفِ ملف موجود.
- لا جلسة تعمل تلقائيًا.
- لا تقرير طويل إذا كان المطلوب قرارًا مختصرًا.
- لا تكرار لنفس المعلومة في أكثر من سجل إلا عند الحاجة.
- الأصل أن `project-control/` هو مركز التنسيق، وليس إنشاء مجلدات إضافية.
