# 1. تحليل المجال (Domain Analysis)

## 1.1 تصنيف نوع التصنيع

| البعد | التصنيف | الشرح |
|:------|:--------|:------|
| **نوع الإنتاج** | **Job Shop / Project-based Manufacturing** | تصنيع حسب الطلب — كل منتج فريد في مواصفاته |
| **حجم الدفعات** | Low-Volume, High-Mix | دفعات صغيرة، تنوع كبير في المنتجات والمواصفات |
| **طبيعة الطلب** | Internal Make-to-Order | مصنع داخلي يخدم مشاريع الشركة — ليس للبيع الخارجي |
| **محرك الإنتاج** | **Engineer-to-Order (ETO)** + **Make-to-Order (MTO)** | منتجات تحتاج رسومات هندسية قبل التصنيع (ETO)، وأخرى متكررة (MTO) |
| **مستوى التكرار** | منخفض—كل منتج له متطلبات فريدة | حتى الأبواب تختلف مقاساتها حسب المشروع |

> **[Research Hint — ثقة: عالية]** هذا التصنيف حاسم. Job Shop يحتاج مرونة عالية — عكس خط الإنتاج الثابت (Mass Production). لا يمكن تصميم مسار واحد rigid لكل المنتجات.

### تأثير Job Shop على النظام

| الخاصية | التأثير |
|:---------|:---------|
| كل منتج له مسار تصنيع مختلف | نظام المراحل يجب أن يكون Customizable (وليس Hardcoded) |
| كل منتج له BOM مختلف | BOM يجب أن يكون Dynamic لكل أمر، وليس Template ثابت |
| الأولويات تتغير يومياً | الـ Scheduling يحتاج مرونة — ليس جدولاً أسبوعياً جامداً |
| رسومات جديدة كل أمر | تكامل مع الـ Drawings ضروري — وليس مجرد خيار إضافي |
| تداخل مشاريع متعددة في وقت واحد | النظام يجب أن يفصل بين الطلبات حسب المشروع |

## 1.2 خصائص المجال المؤثرة على تصميم النظام

### الخصائص التقنية

| الخاصية | الوصف | التأثير على النظام |
|:---------|:------|:-------------------|
| **تعدد وحدات القياس** | طن، كغم، متر، مم، حبة، لتر (دهان) | الحقول: quantity + uom_id (جدول وحدات قياس منفصل) |
| **الأوزان الثقيلة** | الهياكل المعدنية تصل لعشرات الأطنان | حقل weight_kg على مستوى الأمر و القطعة |
| **المواد الخام** | ألواح حديد، مواسير، كيماويات، مهمات لحام | حاجة لمخزون Basic في Phase 3 |
| **مخططات فنية (DXF/PDF)** | كل قطعة لها رسمة مع رقم مراجعة | حقل drawing_number, drawing_revision + رفع ملف |
| **عدد القطع في الأمر** | الهيكل قد يكون 5 قطع أو 500 قطعة | الـ Piece-level tracking يتصرف حسب الحجم |
| **جودة اللحام** | يحتاج فحص بصري + أحياناً اختبارات غير اتلافية | QC يسجل نوع الفحص ونتيجته |
| **الدهان الصناعي** | تحضير سطح (قذف رمل) + طبقات دهان | تسجيل عدد الطبقات ونوع الدهان |

### الخصائص التشغيلية

| الخاصية | الوصف | التأثير |
|:---------|:------|:---------|
| **مقاولين خارجيين (Subcontracting)** | بعض العمليات تُعهد لورش خارجية (قص بالبلازما، الخراطة) | حقل is_subcontract + اسم المقاول + تكلفة |
| **إعادة العمل (Rework)** | عيوب لحام، أبعاد غير صحيحة | Rework Order مرتبط بالأمر الأصلي |
| **التجميع في الموقع** | بعض التصنيع يتم في موقع المشروع وليس في الورشة | حقل location: ورشة / موقع |
| **مواد واردة من المشروع** | أحياناً العميل يورّد الحديد والمصنع يشغّل فقط | حقل material_source: internal / customer-supplied |
| **طلبات عاجلة (Rush Orders)** | مدير المشروع يتصل: "هذا طارئ — قدمه على الكل" | Priority + Rush flag + Override scheduling |
| **تعدد أولويات متزامن** | المصنع يخدم 3-4 مشاريع في وقت واحد | الفلترة حسب المشروع + إمكانية إيقاف/استئناف أوامر |

## 1.3 المصطلحات الأساسية (Domain Glossary)

> هذه المصطلحات ليست أكاديمية — ستتحول إلى جداول وحقول في قاعدة البيانات.

| المصطلح | المعنى | المرجع في النظام |
|:--------|:-------|:-----------------|
| **Production Order (أمر تصنيع)** | المستند الأساسي — طلب تصنيع منتج/مجموعة | الجدول الرئيسي `production_orders` |
| **Work Order (أمر تشغيل)** | أمر لمرحلة محددة في التصنيع | `work_orders` أو حقل مرحلة داخل `production_orders` |
| **Shop Order** | أمر تشغيل داخل الورشة (مستوى أكثر تفصيلاً) | مرادف لـ Work Order في هذا السياق |
| **Routing / Route Card (بطاقة المسار)** | تسلسل المراحل لقطعة معينة | `routing` + `routing_steps` جداول |
| **BOM (Bill of Materials)** | قائمة المواد المطلوبة — هيكل هرمي | `bom` + `bom_items` |
| **Piece Mark (علامة القطعة)** | معرف فريد للقطعة حسب الرسمة (B1, C2, F3...) | `pieces.piece_mark` |
| **WIP (Work In Process)** | القطع تحت التصنيع ولم تكتمل بعد | فلتر: `status NOT IN (delivered,closed)` |
| **Job Ticket / Traveler (بطاقة العمل)** | وثيقة ترافق القطعة — تسجل العمليات والتوقيعات | شاشة Piece Tracking تجمع كل البيانات |
| **Nesting (تقدير القطع)** | ترتيب القطع على ألواح الحديد لتقليل الهدر | يحتاج برنامج CAD متخصص — خارج نطاق النظام حالياً |
| **Cut Plan (خطة القص)** | جدول يحدد أبعاد القص لكل قطعة | يمكن إضافته كحقل نصي/ملف في الأمر |
| **Rework (إعادة عمل)** | تصنيع قطعة من جديد بعد فشل الفحص | `rework_orders` مرتبط بـ `production_orders` |
| **NCR (Non-Conformance Report)** | تقرير عدم مطابقة للمواصفات | يمكن استبداله بـ QC result + notes في MVP |
| **Subcontracting (تصنيع خارجي)** | إسناد عملية لجهة خارجية | `production_orders.is_subcontract` + `subcontractor` |
| **Kitting (تجهيز المواد)** | تجهيز مواد أمر قبل بدء التصنيع | عملية يدوية في MVP — مؤتمتة في Phase 3 |

## 3.3 سيناريوهات خاصة

### السيناريو 1: تصنيع خارجي (Subcontracting)

```
أمر → تعليم "تنفيذ خارجي" ← تسجيل بيانات المقاول
→ إرسال القطع للمقاول ← تسجيل تاريخ الإرسال
→ عند العودة ← استلام ← فحص QC ← يكمل المسار الطبيعي
```

> **[Hint — ثقة: متوسطة]** شائع في الورش الصغيرة — المقاولون يقومون بالقص بالبلازما أو الخراطة CNC. النظام يحتاج حقل `is_subcontract` + `subcontractor_name` + تواريخ الإرسال والاستلام.

### السيناريو 2: طلب عاجل (Rush Order)

```
أمر جديد → علامة "عاجل" + سبب (مثال: عطل في موقع المشروع)
→ يظهر في أعلى قائمة الورشة بلون أحمر
→ المشرف يقرر إيقاف أمر آخر لتسريع العاجل
```

> **[Hint — ثقة: عالية]** آلية الـ Rush شائعة جداً في الورش المعدنية. لا تجعل الجدولة جامدة — العاجل يتجاوز الكل.

### السيناريو 3: مواد من العميل (Customer-Supplied Material)

```
أمر → حقل "مصدر المواد: العميل" ← تسجيل كمية المواد الواردة
→ التصنيع باستخدام مواد العميل ← الفاتورة للمصنع: أجرة تصنيع فقط
```

> **[Hint — ثقة: متوسطة]** بعض مشاريع المقاولات، العميل يورّد الحديد والمصنع يشغّل فقط. هذا يؤثر على حساب التكلفة وطريقة الفوترة.

### السيناريو 4: تجميع في الموقع (On-Site Assembly)

```
أمر → حقل "موقع التصنيع: موقع المشروع"
→ القطع تُصنع في الورشة ثم تُنقل للموقع للتجميع النهائي
→ النظام يتتبع: تصنيع في الورشة + تجميع في الموقع
```

> **[Hint — ثقة: متوسطة]** التجميع النهائي أحياناً يكون في موقع المشروع (للهياكل الكبيرة). يمكن التعامل معه كمرحلة إضافية في المسار.

## 4.4 الأدوار والصلاحيات المقترحة

| الدور | الصلاحيات | يرى فقط | ملاحظات |
|:------|:----------|:--------|:--------|
| **مدير مصنع (Admin)** | قراءة، إنشاء، تعديل، حذف، تقارير، إعدادات | كل الأوامر | لا حدود |
| **مهندس إنتاج** | قراءة، إنشاء، تعديل (بدون حذف)، مراجعة QC | كل الأوامر | يراجع الرسومات ويوزع العمل |
| **مشرف ورشة** | قراءة، تعديل حالة الأمر، تخصيص عمال، أولويات | كل الأوامر النشطة | يدير سير العمل يومياً |
| **عامل ورشة** | بدء/إيقاف/إكمال المرحلة، إضافة ملاحظات | أوامره الموكلة فقط | أبسط واجهة — 3 أزرار |
| **مستلم (Delivery)** | تأكيد تسليم، إدخال ملاحظات | الأوامر الجاهزة فقط | لا يرى التصنيع الداخلي |

> **[Research Hint — ثقة: عالية]** قاعدة ذهبية: **كلما قلّ ما يراه العامل، قلّ خطؤه.** العامل يحتاج فقط: "ما عملي اليوم؟" → يبدأ → يوقف → يكتب ملاحظة.

## 5.1 أفضل نهج للبدء — Phased Rollout

### لماذا التقسيم المرحلي؟

| # | المخاطرة | التأثير | الاحتمال | مستوى الخطورة | استراتيجية التخفيف |
|:-:|:---------|:--------|:--------:|:-------------:|:-------------------|
| R1 | **مقاومة العمال للنظام** | عدم استخدامه أو استخدامه بشكل خاطئ — فشل التبني | متوسط | 🔴 **عالي** | تدريب عملي على التابلت — ابدأ بميزة واحدة تفيدهم — "بطل تغيير" في الورشة |

| R3 | **تعقيد المرونة (Flexibility Trap)** | كل منتج له مسار مختلف — صعوبة في التصميم والتطوير | عالي | 🟡 **متوسط** | لا تحاول تغطية كل سيناريو في MVP — ابدأ بمسارات شائعة وأضف Special Cases لاحقاً |

## 5.3 نصائح للتبني الناجح (Adoption Strategy)

### ✅ افعل — Do

| # | النصيحة | التفصيل |
|:-:|:---------|:---------|
| 1 | **ابدأ بأصغر فائدة مرئية** | أول أسبوعين: أدخل 5 طلبات حقيقية في النظام وتابعها. مشرف الورشة يرى بنفسه "أنا أعرف أين كل طلب" → يقتنع |
| 2 | **اجعل واجهة الورشة بسيطة جداً** | 3 أزرار: Start (أخضر) → Pause (أصفر) → Complete (أزرق). لا قوائم منسدلة. لا حقول إجبارية كثيرة |
| 3 | **درّب "بطل التغيير"** | اختر شخصاً متحمساً في الورشة — دربه أولاً — يعلّم الباقين — يصبح مرجعاً |
| 4 | **استخدم التابلت/الجوال** | الـ UI يجب أن يكون Mobile-first. العامل ليس لديه وقت أو رغبة في الجلوس أمام كمبيوتر |
| 5 | **احتفل بالانتصارات الصغيرة** | بعد أول أسبوع: "هذا الأسبوع أنجزنا 10 أوامر في النظام — ما كان ممكن نعرفهن قبل النظام" |
| 6 | **اربط النظام بحوافز** | العمال الذين يسجلون بيانات دقيقة — تقدير أو مكافأة بسيطة |
| 7 | **فترة تشغيل موازي** | أسبوعان: الطلبات تدخل في النظام + الواتساب كالمعتاد — طمأنة للجميع |
| 8 | **تحديثات قصيرة متكررة** | كل أسبوع — تحديث بسيط بناءً على ملاحظات العمال — "النظام يسمعنا" |

### ❌ لا تفعل — Don't

| # | التحذير | السبب |
|:-:|:---------|:------|
| 1 | **لا تبدأ بكل الميزات معاً** | Phase 1 فقط — لا تخلط QC أو المخزون في البداية — يشتت الجميع |
| 2 | **لا تجعل النظام يعقد حياة العامل** | إذا كان تسجيل مرحلة يأخذ أكثر من 10 ثوانٍ — راجع التصميم |
| 3 | **لا تربط بمحاسبة أو ERP فوراً** | أول شهرين: ركز على تتبع التصنيع فقط. المحاسبة تأتي لاحقاً |
| 4 | **لا تضف تقارير معقدة قبل نضج البيانات** | WIP Age و Rework Rate يحتاجان 3-6 أشهر بيانات — قبل ذلك أرقام غير ذات معنى |
| 5 | **لا تفرض النظام بقوة** | "الكل يستخدم النظام من يوم الأحد" ← فشل ذريع. ابدأ بمستخدم واحد متحمس ثم توسع |
| 6 | **لا تتجاهل ملاحظات العمال** | "هذا الزر ما زبط" ← إذا تجاهلتها → يفقدون الثقة ← لا يستخدمون النظام |
