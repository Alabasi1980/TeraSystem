# تقييم DomainExpertAgent — تقرير للمالك و TeraSystemEvolutionAgent

**تاريخ التقرير:** 2026-07-06  
**المراجع:** TCEA (TeraClientEngagementAgent — مستشار)  
**الغرض:** تقييم قدرة DomainExpertAgent على تحليل بحوث المشتريات في أنظمة ERP وإنتاج هيكل دراسة متكامل.

---

## 1. الخلفية

مشروع شركة الحوت للمقاولات يتطلب بناء دراسة استشارية شاملة لقسم المشتريات، مطابقة بنسبة 90% لأفضل ممارسات أنظمة ERP العالمية (SAP MM، Oracle Procurement، Dynamics 365). تم حتى الآن إنتاج 4 تقارير بحثية (R01–R03, R08) بواسطة DomainResearchAgent.

الخطوة التالية تتطلب **تحليل هذه البحوث وتصنيفها** وبناء **هيكل الدراسة النهائي** — وهذا هو دور DomainExpertAgent المفترض.

### 2.2 ما يمكنه فعله حالياً ✅

| الإمكانية | التفاصيل |
|:----------|:----------|
| يقرأ Domain Research Report(s) | يستطيع استيعاب تقارير بحثية متعددة |
| يصنف التوصيات | Include now / Recommended / Defer / Out of Scope / Needs User Decision |
| يحلل مجالاً معيناً | يحول المعلومات الخام إلى تحليل منظم |
| ينتج Domain Intelligence Report | تقرير ذكي واحد شامل |
| يُستدعى مباشرة من TCEA | صلاحية خاصة بموجب A.7.1 |

### 2.3 حدوده الحالية ❌

| القيد | المشكلة لهذا المشروع |
|:------|:---------------------|
| **التصنيف موجه لتطوير برمجي** (MVP)، وليس لبناء دراسة استشارية | هذا المشروع ليس برمجياً — الدراسة ليست تطبيقاً. فئات التصنيف لا تناسب "أقسام دراسة وثائقية" |
| **ينتج تقريراً واحداً فقط** (Domain Intelligence Report) | نحتاج مخرجات متعددة: هيكل دراسة + تحليل فجوات + بنك مصطلحات |
| **لا يبني هيكلاً وثائقياً (Document Outline)** | لا يستطيع إنتاج جدول محتويات دراسة أو هيكل فصول |
| **تصنيفه يركز على Scope** (ما يدخل/لا يدخل في النطاق البرمجي) | هذا المشروع يحتاج تصنيفاً معرفياً (Core / Supporting / Contextual) |
| **لا يدعم التصنيف الهرمي متعدد المستويات** | Procurement Study تحتاج فصولاً وأقساماً وأجزاء فرعية |
| **لا يوجد له ملف Agent فعلي** | لم يُولد بعد — مجرد تعريف في TeraSubAgents.md |
| **لم يُختبر فعلياً من قبل** | هذا أول مشروع يستدعيه |
| **المخرجات بالعربية مع المصطلحات الإنجليزية** | غير مؤكد إذا كان يدعم ذلك أصلاً |

## 3. تقييم القدرة على تنفيذ المهمة المطلوبة

### المهمة المطلوبة منه
```
استلام 4 تقارير بحثية (R01, R02, R03, R08) →
تحليلها واستخراج العناصر المعرفية →
تصنيفها حسب نوع المعلومة (عملية/هيكلية/تنظيمية) →
بناء هيكل دراسة شامل للمشتريات (فصول + أقسام + أجزاء) →
تحديد الفجوات التي تحتاج بحوثاً إضافية →
إنتاج Domain Intelligence Report مفصل
```

### التقييم

| المعيار | التقييم | الشرح |
|:--------|:-------:|:------|
| قراءة التقارير | ✅ ممتاز | المقروئية مضمونة |
| تحليل المحتوى | ✅ جيد | يمكنه تحليل واستخراج العناصر |
| تصنيف المعلومات | ⚠️ محدود | التصنيف الحالي لا يناسب دراسة استشارية |
| بناء هيكل وثائقي | ❌ غير موجود | لا يدعم هذه الميزة حالياً |
| تحديد الفجوات | ⚠️ جزئي | لا توجد قاعدة واضحة لتحديد الفجوات |
| تكامل مع DomainResearchAgent | ⚠️ غير محدد | لا يوجد Protocol تكامل واضح بينهما |
| العمل بدون إشراف | ⚠️ بحاجة لتوجيه | سيحتاج تعليمات دقيقة جداً في كل استدعاء |

**الخلاصة: DomainExpertAgent الحالي يمكنه المساعدة جزئياً ولكنه محدود — يحتاج تحديثاً ليناسب هذا النوع من المشاريع.**

## 4. التوصيات للتطوير

### 4.1 تحديث مقترح لـ TeraSubAgents.md §6.13

أقتراح الإضافات التالية لتعريف DomainExpertAgent:

```text
### مخرجات إضافية (للمشاريع الاستشارية غير البرمجية):
- Domain Knowledge Structure: هيكل وثائقي هرمي (Table of Contents + Chapter Outline)
- Gap Analysis Report: فجوات المعرفة التي تحتاج بحثاً إضافياً
- Cross-Reference Map: علاقات الترابط بين المواضيع

### تصنيفات إضافية للمعلومات:
- Core Process (عملية أساسية)
- Supporting Activity (نشاط مساند)
- Structural Element (عنصر هيكلي/تنظيمي)
- Contextual Knowledge (معلومة سياقية)
- Cross-Cutting (عرضي — يمس عدة أقسام)

### عند بناء هيكل دراسة:
1. قراءة جميع Domain Research Reports المتاحة
2. استخراج العناصر المعرفية وتصنيفها
3. بناء هيكل هرمي (فصول ← أقسام ← أجزاء)
4. تحديد الفجوات (Gaps) مع اقتراح أولويات البحث
5. إنتاج Knowledge Structure Report بدلاً من (أو بالإضافة إلى) Domain Intelligence Report
```

### 4.2 إنشاء ملف عميل فعلي

يحتاج DomainExpertAgent إلى ملفه الخاص:

```text
.opencode/agents/domain-expert-agent.md
```

ليكون قابلاً للاستدعاء الفعلي من TCEA أو TeraAgent.

### 4.3 تحسين التعاون بين DomainResearchAgent و DomainExpertAgent

المقترح: عندما يستدعي TCEA كليهما، يتم تمرير مخرجات ResearchAgent مباشرة كمدخلات ExpertAgent في سلسلة (Pipeline) واحدة:

```
TCEA → DomainResearchAgent → DomainResearchReport
     → DomainExpertAgent (يقرأ التقرير) → DomainIntelligenceReport + KnowledgeStructure
     → TCEA يوثق في مجلد العميل
```

## 6. ماذا أفعل في هذه الأثناء؟

لعدم تعطيل المشروع، أقترح:

1. **نواصل البحوث المتبقية (R04–R15)** يدوياً عبر DomainResearchAgent — هذا لا يحتاج ExpertAgent
2. في نفس الوقت، DomainExpertAgent يُطوَّر ليلحق بنا
3. عندما يكتمل التطوير، يبدأ DomainExpertAgent تحليل كل البحوث دفعة واحدة وإنتاج الهيكل النهائي

**هذا يعني: لا تأخير في المشروع — نعمل على المسارين بالتوازي.**
