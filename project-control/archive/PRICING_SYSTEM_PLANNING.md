**المُعد:** TeraSystemEvolutionAgent  
**الحالة:** تخطيط أولي — قبل أي بناء  
**الغرض:** توضيح الفكرة، استعراض الخيارات، تمهيد القرار

---

## مقدمة

هذا المستند هو **مرحلة التخطيط فقط**. ليس هناك أي كود أو ملفات نظامية تم بناؤها بعد.  
الهدف هو مساعدتك (Majed) في فهم:

1. ما هو نظام التسعير البرمجي أصلاً؟
2. ما هي الخيارات المتاحة؟
3. كيف يبدو السوق الأردني؟
4. كيف نبني نظاماً عادلاً ومناسباً لـ Tera؟
5. ما هي الأسئلة التي تحتاج إجابات قبل البناء؟

بعد قراءة هذا المستند ومناقشته، سنقرر معاً **ماذا نبني**، ثم أنتج  
`SYSTEM_CHANGE_PROPOSAL` لتعديل `TeraClientEngagement.md` وإضافة النظام.

## الجزء الأول: ما هو نظام التسعير البرمجي؟

### المشكلة الأساسية

تطوير البرمجيات ليس مثل شراء منتج جاهز.  
كل تطبيق فريد من حيث:
- عدد الشاشات
- تعقيد الميزات
- التكاملات الخارجية
- عدد المستخدمين
- التصميم
- الوقت

**لذلك:** لا يمكن أن يكون هناك سعر ثابت لكل التطبيقات.

### نماذج التسعير الشائعة عالمياً

| النموذج | الشرح | يناسب |
|---------|-------|-------|
| **Fixed Price** (سعر ثابت) | تتفق على سعر نهائي للمشروع كامل | مشاريع صغيرة، النطاق واضح تماماً |
| **Time & Materials** (ساعة/يوم) | تدفع حسب الوقت الفعلي | مشاريع كبيرة، النطاق متغير |
| **Hybrid** (مختلط) | Discovery بسعر ساعة → تنفيذ بسعر ثابت | الحالة الأكثر شيوعاً للمشاريع المتوسطة |
| **Value-Based** (على أساس القيمة) | السعر يعتمد على القيمة التي سيحققها التطبيق للزبون | مشاريع استراتيجية، نادر |
| **Feature-Based** (على أساس الميزات) | كل ميزة لها سعر — تجمع الميزات = السعر الكلي | شفاف، عادل، يناسب Tera |

### كيف تُقدر التكلفة فعلياً؟

أشهر الطرق العالمية:

1. **Bottom-Up Estimation (تقدير تصاعدي)**
   - تحلل المشروع إلى مهام صغيرة
   - تقدر الوقت لكل مهمة
   - تجمع الكل → السعر
   - *الأدق لكنه يحتاج وقت*

2. **Function Point Analysis (تحليل نقاط الوظائف)**
   - تحسب عدد: المدخلات، المخرجات، الاستعلامات، الملفات، الواجهات
   - كل نقطة لها وزن
   - *دقيق لكنه يحتاج خبير*

3. **Comparative Estimation (تقدير مقارن)**
   - تقارن المشروع بمشاريع سابقة مماثلة
   - تعدل لأعلى/لأسفل حسب الفروقات
   - *سريع، مناسب لـ Tera بعد أول 5-10 مشاريع*

4. **T-Shirt Sizing (تقدير بالأحجام)**
   - تصنف المشاريع إلى: S, M, L, XL, XXL
   - لكل حجم نطاق سعري
   - *بسيط، مناسب للمرحلة الأولى*

## الجزء الثالث: مبادئ التسعير العادل — توصياتي

بناءً على طلبك في عدم استغلال الزبون، هذه المبادئ المقترحة:

### المبدأ 1: السعر حسب الحجم الفعلي
```
تطبيق بشاشتين + قاعدة بيانات + تسجيل دخول ≠ تطبيق بـ 20 شاشة + ERP + 3 تكاملات
الفرق يجب أن ينعكس في السعر.
```

### المبدأ 2: الشفافية الكاملة
```
الزبون يعرف بالضبط:
- كم سعر كل جزء من التطبيق
- لماذا هذا السعر
- ماذا لو أراد زيادة أو تغيير
لا توجد أرقام مبهمة.
```

### المبدأ 3: Margin معقول — ليس الأغلى ولا الأرخص
```
السوق الأردني متوسط: أنت لا تريد أن تكون أرخص شركة (قلة جودة)
ولا أغلى شركة (استغلال).
كن في الربع المتوسط: جودة عالية بسعر عادل.
```

### المبدأ 4: التسعير لا يعتمد على "قدرة الزبون على الدفع"
```
السعر مرتبط بالعمل، لا بالعميل.
العميل الغني والعميل المقتدر يدفعان نفس السعر لنفس التطبيق.
```

### المبدأ 5: هامش ربح صحي — لأن البقاء يعتمد عليه
```
السعر العادل لا يعني السعر الأقل.
السعر العادل = تكلفة + جهد + خبرة + هامش ربح معقول (15-25%).
```

## الجزء الرابع: نموذج التسعير المقترح لـ Tera

### المكونات الأساسية للنظام

```
نظام التسعير = 
  1. مصفوفة التسعير (Pricing Matrix) ← القاعدة الأساسية
  2. حاسبة الأسعار (Pricing Calculator) ← TCEA يستخدمها
  3. قواعد التخصيص (Customization Rules) ← ميزات إضافية
  4. قالب عرض السعر (Quotation Template) ← المخرجات
```

### 1. مصفوفة التسعير — Pricing Matrix المقترحة

هذه هي **اللبنة الأساسية**. تحدد سعر كل مكون في التطبيق.

#### أ. سعر الأساس — Base Price (حسب نوع التطبيق)

| النوع | التعقيد | Base Price (JOD) |
|-------|---------|-----------------|
| موقع تعريفي / Landing Page | بسيط | 300 - 600 |
| موقع مؤسسي (Company Website) | متوسط | 600 - 1,200 |
| متجر إلكتروني صغير | متوسط | 1,200 - 2,500 |
| تطبيق موبايل (منصة واحدة) | متوسط | 1,500 - 3,500 |
| Web App / SaaS صغير | متوسط | 2,000 - 5,000 |
| ERP / نظام إداري | معقد | 3,000 - 8,000 |
| تطبيق متكامل (موبايل + ويب + API) | معقد جداً | 5,000 - 15,000 |

#### ب. وحدات إضافية — Add-on Modules

> المهم الآن هو **الهيكل** وليس **الأرقام النهائية**.

### 2. حاسبة الأسعار — آلية العمل داخل TCEA

```

يملأ حقول التقييم التالية:
```

| الحقل | المصدر من Discovery |
|-------|-------------------|
| نوع التطبيق | مناقشة مع الزبون |
| عدد الشاشات | وصف الميزات |
| عدد أدوار المستخدمين | Q6.1 - Q6.2 |
| التكاملات المطلوبة | Q4.4 |
| عدد اللغات | Q2.7 |
| تقارير أم لا؟ | Q3.4 - Q3.5 |

**ثم:**  
يجمع TCEA السعر من المصفوفة ← ينتج مسودة عرض سعر ← Majed يراجع ← يعتمد.

## الجزء السادس: أسئلة تحتاج إجابتك قبل البناء

هذه أسئلة **استراتيجية** أنت وحدك من يقررها:

### س1: كيف تحدد الـ Margin؟
```
هل تريد:
أ) هامش ربح منخفض (10%) ← أسعار أقل ← عملاء أكثر ← تنافس عالي
ب) هامش ربح متوسط (20%) ← أسعار عادلة ← توازن
ج) هامش ربح مرتفع (30%) ← ربحية أعلى ← عملاء أقل

ملاحظة: "عادل" لا يعني الأرخص.
السعر العادل = يغطي التكاليف + يضمن جودة + يحقق ربحاً معقولاً.
```

### س2: التسعير حسب التعقيد أم حسب الوقت؟
```
أ) Feature-Based (موصى به) — كل ميزة بسعرها ← شفاف، عادل
ب) Hourly Rate — سعر للساعة ← مرن لكن غير متوقع للزبون
ج) Mixed — Discovery بالساعة، تنفيذ بسعر ثابت ← أفضل حل وسط
```

### س3: هل تسعّر الصيانة ضمنياً أم صراحةً؟
```
أ) ضمن السعر الكلي (ضمان + صيانة أول 3 شهور)
ب) الصيانة بند مستقل (X JOD/شهر)
ج) الاثنان معاً (فترة مجانية ثم اشتراك)
```

### س4: هل تسعّر الاستضافة؟
```
أ) يتحملها الزبون مباشرة (نحن نوصي فقط)
ب) نؤمنها ونضيف هامشاً بسيطاً
ج) ضمن باقة متكاملة
```

### س5: كيف تتعامل مع طلبات التغيير أثناء العمل؟
```
أ) كل تغيير خارج النطاق → سعر إضافي حسب المصفوفة
ب) أول X تغييرات مجاناً → بعدها يسعّر
ج) حسب الوقت الفعلي (Time & Materials)
```

## خلاصة — رسالة مباشرة لك يا Majed

أنت لست بحاجة لمعرفة كل شيء عن التسعير مسبقاً.

**ما تحتاجه الآن فقط هو:**
1. فهم الخيارات المتاحة (قدمتها لك أعلاه)
2. اختيار الاتجاه (أجب على الأسئلة الخمسة في الجزء السادس)
3. وضع أرقام تقريبية (يمكن تعديلها لاحقاً)

أنا سأتكفل بالباقي: بناء النظام، تحديث TCEA، إنتاج القوالب، والتأكد من عدم حدوث تضخم أو تعقيد غير ضروري.

> **تذكر:** أول 5 مشاريع ستكشف لك الثغرات في أي نظام تسعير تبنيه.  
> لذلك المهم الآن هو **البدء بهيكل بسيط قابل للتعديل**، وليس نظاماً كاملاً من أول يوم.
