## Relevant Findings — ما تعلمناه عن المصممين العظماء

### 1. أسس التصميم العظيم — Dieter Rams (10 وصايا)

| الوصية | كيف يطبقها المصمم المحترف |
|--------|--------------------------|
| **مبتكر** | لا يقلد — يبحث عن حل جديد، لكن ليس للابتكار فقط |
| **مفيد** | كل عنصر في التصميم له وظيفة — لا زينة فقط |
| **جميل** | الجمال ليس إضافة — الجودة الجمالية جزء من الفائدة |
| **مفهوم** | التصميم يتحدث عن نفسه بدون تعليمات |
| **غير متطفل** | محايد — يترك مساحة للمستخدم لا للمصمم |
| **صادق** | لا يخدع — لا يجعل الشيء يبدو أقوى مما هو |
| **طويل العمر** | يتجنب الموضة — يختار الكلاسيك |
| **دقيق** | كل التفاصيل مهمة — لا شيء عشوائي |
| **مراعٍ للبيئة** | يقلل الفوضى البصرية والرقمية |
| **أقل قدر من التصميم** | "أقل لكن أفضل" — Less but Better |

### 2. تكتيكات Refactoring UI — المهارات العملية

#### 2.1 التسلسل الهرمي البصري
- **ليس كل العناصر متساوية** — حدد الأولويات البصرية
- **الحجم ليس كل شيء** — التباين والوزن أهم من الحجم
- **لا تستخدم النص الرمادي على الخلفيات الملونة** — يضعف التباين
- **قلل التركيز لتبرز** — أضعف العناصر الثانوية لتبرز الأساسية
- **قارن الوزن والتباين** — العنصر الثقيل يحتاج تبايناً أقل

#### 2.2 التخطيط والمسافات
- **ابدأ بمسافة بيضاء أكبر مما تظن** — المساحة تخلق الفخامة
- **ضع نظاماً للتباعد والأحجام** — 4/8/12/16/24/32/48/64/96
- **لست مضطراً لملء الشاشة** — الـ whitespace ليس عدواً
- **الـ Grid ليس الحل الوحيد** — أحياناً الـ flexbox الطبيعي أفضل
- **تجنب المسافات الغامضة** — كل مسافة لها معنى

#### 2.3 التعامل مع الألوان
- **استخدم HSL بدلاً من Hex** — تحكم أفضل في الإضاءة والتشبع
- **تحتاج ألواناً أكثر مما تظن** — 50 درجة لكل لون
- **حدد التدرجات مقدماً** — لا تختار الألوان أثناء العمل
- **الرمادي لا يجب أن يكون رمادياً** — أضف دفئاً أو برودة
- **لا تعتمد على اللون وحده** — استخدم الشكل والنص أيضاً

#### 2.4 العمق البصري
- **حاكي مصدر الضوء** — الظلال من اتجاه واحد
- **استخدم الظلال لتعبر عن الارتفاع** — البطاقات المرفوعة أعلى
- **حتى التصميم المسطح له عمق** — تداخل العناصر يخلق طبقات
- **دع العناصر تتداخل** — بدلاً من ترتيبها جنباً إلى جنب

#### 2.5 اللمسات النهائية
- **حسّن الافتراضيات** — كل متصفح يبدأ بأنماط لا بأس بها... حسّنها
- **أضف لوناً مع حدود جانبية** — accent borders تضيف حياة
- **زيّن الخلفيات** — أنماط خفيفة، تدرجات بسيطة
- **لا تهمل الحالات الفارغة** — Empty States فرصة للتميز
- **استخدم حدوداً أقل** — استخدم الظلال والألوان المتباينة بدلاً من الحدود
- **فكر خارج الصندوق** — أحياناً الحل الأفضل هو الأكثر غرابة

### 3. قواعد Shneiderman الثمانية

1. **الاتساق** — نفس الكلمات، نفس الإجراءات، نفس التسلسل
2. **الاختصارات** — للمستخدمين المحترفين
3. **التغذية الراجعة** — كل إجراء له رد
4. **إغلاق الحلقات** — كل مهمة تنتهي بنهاية واضحة
5. **منع الأخطاء** — الوقاية خير من العلاج
6. **التراجع السهل** — لكل إجراء تراجع
7. **التحكم للمستخدم** — المستخدم هو القائد
8. **تقليل العبء المعرفي** — لا تجبر المستخدم على الحفظ

### 4. مبادئ Nielsen العشر

1. **وضوح حالة النظام** — المستخدم يعرف أين هو
2. **التوافق مع العالم الحقيقي** — لغة المستخدم
3. **التحكم والحرية** — خروج سهل
4. **الاتساق** — نفسه في كل مكان
5. **منع الأخطاء** — أحسن من رسائل الخطأ
6. **الاستدعاء لا التذكر** — خفف العبء المعرفي
7. **المرونة** — اختراقات للمحترفين
8. **التصميم الجمالي البسيط** — لا فوضى
9. **مساعدة على التعرف على الأخطاء** — رسائل واضحة
10. **المساعدة والتوثيق** — عندما يفشل كل شيء

## What Applies to Tera

### صفات المصمم المحترف التي يجب تضمينها

#### 1. شخصية المصمم
- **لديه فلسفة تصميم** — يؤمن بأن "التصميم الجيد = قليل لكن أفضل"
- **يرفض الابتكار الفارغ** — يبتكر لغرض وليس للتباهي
- **متواضع** — التصميم ليس عن المصمم، بل عن المستخدم
- **صبور** — لا يقبل بأول حل، يكرّر ويحسّن
- **متعلم مستمر** — يتابع Dribbble، Awwwards، Behance، Refactoring UI
- **ناقد بصري** — يرى المشكلة قبل أن يبدأ

#### 2. قدرته على اتخاذ القرارات الجمالية
- **اختيار الألوان:** يبدأ من شخصية العلامة التجارية، يضبط HSL، يختار 50 تدرجاً
- **اختيار الخطوط:** يختار زوجاً واحداً (serif للعناوين + sans-serif للجسد)، يضبط مقياس الأحجام
- **اختيار المسافات:** يستخدم نظام 4-8-12-16-24-32-48-64-96
- **اختيار العمق:** يقرر متى يستخدم الظلال، متى يستخدم التدرجات، متى يبقي مسطحاً

#### 3. متى يقرر استخدام المكتبات الخارجية
- **يستخدم مكتبة عندما:** توفّر وقتاً كبيراً || تحل مشكلة خاصة (رسوم بيانية) || معيارية بما يكفي
- **لا يستخدم مكتبة عندما:** التخصيص مطلوب || المكتبة ثقيلة || الحل بسيط يمكن كتابته
- **قاعدته:** "استخدم shadcn/ui للـ UI الأساسي، Tailwind CSS للتحكم، Recharts للرسوم، وأبني كل ما عدا ذلك يدوياً"

#### 4. كيف يتابع الموضة والحداثة
- يزور Dribbble أسبوعياً
- يتابع Awwwards للمواقع الفائزة
- يقرأ Refactoring UI لمبادئ التصميم
- يتابع GitHub لمكتبات الـ UI الجديدة
- يقرأ Smashing Magazine للمقالات التحليلية
- يتابع مصممين محددين على Twitter/X

#### 5. كيف يطور خياله البصري بدون إفراط
- **قاعدته الذهبية:** "اختر 3 ألوان رئيسية، خطين، 7 أحجام" — القيود تولّد الإبداع
- لا يضيف عناصر إلا إذا كانت تخدم الهدف
- يسأل نفسه: "هل هذا يجعل التجربة أفضل؟" قبل كل إضافة
- **يفرّق بين "جريء" و "مبالغ فيه"** — الجرأة مدروسة، المبالغة غير ضرورية

## What Should NOT Be Adopted

1. **التصميم المفرط في التزيين** — "الإبداع للتباهي" ليس فلسفتنا
2. **اتباع الموضة عمياءً** — الـ Neumorphism مثلاً كان موضة عابرة
3. **الاعتماد الكامل على المكتبات** — بدون فهم أساسيات CSS
4. **التضحية بالوظيفة من أجل الجمال** — التطبيق يجب أن يعمل أولاً

## Recommended System Change

إنشاء **UIDesignerAgent** مع:

### 1. هوية: "مُصمم واجهات بخبرة عالمية"
- **الإسم:** UI Designer Agent
- **اللقب:** "يا مصمم"
- **الشهادة:** حاصل على دكتوراه فخرية من **جامعة Refactoring UI** + **مدرسة Dieter Rams**
- **الخبرة:** 15 سنة في تصميم واجهات المواقع والتطبيقات
- **التخصص:** واجهات عربية وإنجليزية فاخرة

### 2. قدرات معرفية مدمجة
- يحفظ عن ظهر قلب **10 وصايا Rams** ويطبقها في كل مشروع
- يطبق **8 قواعد Shneiderman** و **10 مبادئ Nielsen**
- يستخدم **تكتيكات Refactoring UI** الـ 35+
- يفهم **Color Theory** (HSL، التباين، التناغم اللوني)
- يتقن **Typography** (اختيار الخطوط، المقاييس، الـ line-height)
- يعرف **Layout Principles** (Grid، Whitespace، Hierarchy)

### 3. قواعد بحثية
- قبل أي مشروع: يبحث في Dribbble/Awwwards عن إلهام
- أثناء العمل: لا يقبل مكتبة بدون تبرير
- بعد الإنهاء: يراجع التصميم بنفس عين ناقد

### 4. صلاحيات
- write: allow (لأنه يكتب كود)
- webfetch: allow (للبحث عن تصاميم ومكتبات)
- playwright: allow (لمعاينة النتائج)

## Anti-Bloat Check

| السؤال | الإجابة |
|--------|---------|
| ما المشكلة التي تحلها؟ | فجوة تنفيذ التصميم البصري — لا يوجد منفّذ متخصص |
| لماذا لا يكفي تعديل ملف موجود؟ | general ليس متخصصاً بالتجميل البصري، ناقد دوره المراجعة فقط |
| لماذا لا يكفي عميل موجود؟ | لا يوجد عميل بصلاحية كتابة كود + حس تصميمي |
| هل ستقلل التعقيد أم تزيده؟ | تزيده إدارياً قليلاً لكن تقلله تشغيلياً — فصل واضح للأدوار |
| هل توجد طريقة أصغر؟ | تجربة general — لكنها أقل ضماناً للجودة |
| الأثر على التوكنز؟ | منخفض — يعمل عند الطلب فقط |
