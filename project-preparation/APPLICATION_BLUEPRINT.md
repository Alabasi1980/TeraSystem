## 1. Application Overview — نظرة عامة على التطبيق

| الحقل | القيمة |
|-------|--------|
| اسم التطبيق | نظام إدارة طلبات الصيانة الداخلي |

## 4. Proposed Modules / Major Capabilities — الوحدات المقترحة

بناءً على تحليل حزمة التسليم، يُقترح تقسيم التطبيق إلى **4 وحدات رئيسية**:

### الوحدة 1: المصادقة والصلاحيات (Authentication & RBAC)

| المكون | وصف |
|--------|------|
| تسجيل الدخول | اسم مستخدم + كلمة مرور |
| تسجيل الخروج | إنهاء الجلسة |
| إدارة الجلسات | الحفاظ على الجلسة أثناء الاستخدام |
| 3 أدوار صلاحيات | موظف / مسؤول صيانة / مدير إداري مع تقييد الوصول حسب الدور |

**مستوى التعقيد المتوقع:** 🟢 بسيط (Simple)

---

### الوحدة 2: إدارة طلبات الصيانة (Maintenance Request Management)

| المكون | وصف |
|--------|------|
| إنشاء طلب صيانة | القسم، نوع الصيانة، الأولوية، الوصف |
| عرض قائمة الطلبات | عرض جدولي مع إمكانية التصفية (حالة، تاريخ، قسم) |
| صفحة تفاصيل الطلب | معلومات كاملة + سجل التعليقات |
| سير عمل الحالات | جديد ← قيد التنفيذ ← مكتمل / ملغي |
| تحديث الحالة | من مسؤول الصيانة |
| نظام التعليقات | إضافة تعليقات على الطلب من مسؤول الصيانة |
| طلباتي (للموظف) | عرض طلبات الموظف فقط |

**مستوى التعقيد المتوقع:** 🟡 متوسط (Medium)

---

### الوحدة 3: Dashboard الإداري (Admin Dashboard)

| المكون | وصف |
|--------|------|
| بطاقات المؤشرات | عدد الطلبات الجديدة، قيد التنفيذ، المكتملة، الملغية |
| رسم بياني بسيط | توزيع الطلبات حسب النوع أو الحالة |

**مستوى التعقيد المتوقع:** 🟢 بسيط (Simple)

---

### الوحدة 4: إدارة المستخدمين (User Management)

| المكون | وصف |
|--------|------|
| قائمة المستخدمين | عرض + حالة (نشط/معطل) |
| إضافة مستخدم | الاسم، البريد، الدور، القسم، كلمة المرور |
| تعديل مستخدم | تعديل بيانات المستخدم |
| تعطيل/تفعيل | تغيير الحالة دون حذف |

**مستوى التعقيد المتوقع:** 🟢 بسيط (Simple)

## 5. Proposed User Roles / Operational Actors — الأدوار المقترحة

| الدور | العدد | الصلاحيات المقترحة |
|-------|:-----:|--------------------|
| **موظف (Employee)** | ~25 | • إنشاء طلب صيانة جديد<br>• عرض طلباته فقط<br>• عرض تفاصيل طلباته |
| **مسؤول صيانة (Maintenance)** | 2 | • عرض جميع الطلبات<br>• تحديث حالة أي طلب<br>• إضافة تعليقات<br>• إلغاء طلب إذا لزم |
| **مدير إداري (Admin)** | 1 | • جميع صلاحيات مسؤول الصيانة<br>• Dashboard الإداري<br>• إدارة المستخدمين (إضافة/تعديل/تعطيل)<br>• متابعة عامة |

### ملاحظات على الأدوار

- الأدوار الثلاثة مقترحة كما وردت في حزمة التسليم دون تغيير.
- ليس هناك صلاحية "حذف مستخدم" — فقط تعطيل/تفعيل.
- الموظف لا يعدّل طلبه بعد أن يبدأ مسؤول الصيانة بمعالجته (يحتاج تأكيد).

## 6. Proposed Workflow Shape — شكل سير العمل المقترح

### سير عمل الطلب الرئيسي

```text
[موظف] ينشئ طلباً
       │
       ▼
    ┌──────────┐
    │  جديد    │ ◄─────────────── [مسؤول صيانة] يضيف تعليق
    └────┬─────┘                   (يبقى في نفس الحالة)
         │
         ▼
    ┌──────────┐
    │قيد التنفيذ│ ◄─────────────── [مسؤول صيانة] يحدّث الحالة
    └────┬─────┘                   (من جديد → قيد التنفيذ)
         │
         ├────────────────────┐
         ▼                    ▼
    ┌──────────┐        ┌──────────┐
    │ مكتمل    │        │ ملغي     │
    └──────────┘        └──────────┘
```

### قيود انتقال الحالة المقترحة

| من | إلى | من ينفذ |
|----|-----|---------|
| جديد | قيد التنفيذ | مسؤول صيانة |
| جديد | ملغي | مسؤول صيانة / مدير إداري |
| قيد التنفيذ | مكتمل | مسؤول صيانة |
| قيد التنفيذ | ملغي | مسؤول صيانة / مدير إداري |
| مكتمل | — | (حالة نهائية — لا انتقال منها) |
| ملغي | — | (حالة نهائية — لا انتقال منها) |

### ملاحظات على سير العمل

- الحالات المقترحة هي فقط: **جديد، قيد التنفيذ، مكتمل، ملغي**.
- حالات إضافية مثل "بانتظار قطع/موافقة" أو "مغلق" مؤجلة لمرحلة لاحقة.
- مسؤول الصيانة يمكنه إضافة تعليقات دون تغيير الحالة (يبقى الطلب في حالته الحالية).

## 9. Technical Decision Candidates — مرشحات القرارات التقنية

> ⚠️ **تنبيه:** ما يلي عبارة عن **مرشحات وتوصيات فقط**، وليست قرارات نهائية. القرار التقني النهائي يُتخذ في مرحلة التحضير الرسمي (Project Preparation) بواسطة TeraAgent و SolutionArchitectureAgent.

### 9.1 Backend — لغة البرمجة والإطار

| المرشح | المزايا | الاعتبارات |
|--------|---------|------------|
| **Node.js + Express/NestJS** | • خفيف وسريع<br>• مناسب لتطبيق داخلي بسيط<br>• مجتمع كبير | • يحتاج إدارة JavaScript/TypeScript |
| **C# + ASP.NET Core** | • أداء عال<br>• أمان مدمج<br>• مناسب لأنظمة Windows | • بيئة تطوير أثقل |
| **Python + Django/FastAPI** | • تطوير سريع<br>• مناسب للنماذج الأولية | • أداء أقل في التطبيقات متعددة المستخدمين |

**التوصية المبدئية:** Node.js مع TypeScript — مناسبة لحجم التطبيق ومستوى التعقيد.

### 9.2 Frontend — إطار الواجهة

| المرشح | المزايا | الاعتبارات |
|--------|---------|------------|
| **React + Next.js** | • مرونة عالية<br>• دعم RTL جيد<br>• مجتمع كبير | • يحتاج تكوين إضافي |
| **Vue.js + Nuxt** | • بساطة<br>• منحنى تعلم أقل | • مجتمع أصغر قليلاً |
| **Blazor (إذا كان Backend C#)** | • لغة واحدة (C#)<br>• تكامل سلس مع .NET | • أقل نضجاً في دعم RTL |

**التوصية المبدئية:** React مع Next.js (App Router) — مرن، متوافق مع RTL، ومناسب لتطبيق به 7 شاشات.

### 9.3 قاعدة البيانات

| المرشح | المزايا | الاعتبارات |
|--------|---------|------------|
| **PostgreSQL** | • مجاني وقوي<br>• دعم JSON<br>• مناسب للبيانات العلائقية | • يحتاج إدارة خادم |
| **SQLite** | • لا يحتاج خادم<br>• أبسط نشر<br>• مناسب لـ ~25 مستخدم | • أداء محدود مع التوسع |
| **SQL Server Express** | • مجاني (طبقة محدودة)<br>• مناسب لبيئة Windows | • قيود على حجم DB |

**التوصية المبدئية:** PostgreSQL — توازن جيد بين القوة والبساطة للحجم المتوقع.

### 9.4 الاستضافة

| المرشح | المزايا | الاعتبارات |
|--------|---------|------------|
| **محلية (On-Premise / Local Network)** | • تحكم كامل<br>• لا اشتراكات شهرية | • مسؤولية العميل في الصيانة |
| **سحابية (VPS صغير — مثل Hetzner / DigitalOcean)** | • توفر عال<br>• إدارة أسهل | • تكلفة شهرية ~5-15$ |

**التوصية المبدئية:** محلية (On-Premise) في النسخة الأولى — حسب افتراضات الحزمة، مع ترك الخيار مفتوحاً للانتقال للسحابة لاحقاً.

### 9.5 نمط المصادقة

| المرشح | الوصف |
|--------|-------|
| **Session-based Auth** | جلسات تقليدية مع تخزين آمن لكلمة المرور (bcrypt) |
| **JWT-based Auth** | توكنز ممتدة — مناسبة إذا تطلب API لاحقاً |

**التوصية المبدئية:** Session-based Auth — أبسط وأكثر أماناً لتطبيق داخلي بمستخدمين محدودين.

### 9.6 UI Framework

| المرشح | المزايا |
|--------|---------|
| **Tailwind CSS** | • تحكم كامل بالتصميم<br>• ملفات CSS خفيفة<br>• دعم RTL |
| **Bootstrap 5 + RTL** | • مكونات جاهزة<br>• دعم RTL مدمج |
| **shadcn/ui (مع Tailwind)** | • مكونات قابلة للتخصيص<br>• تصميم حديث |

**التوصية المبدئية:** Tailwind CSS + shadcn/ui — مزيج حديث يسمح ببناء واجهة عربية RTL نظيفة.
