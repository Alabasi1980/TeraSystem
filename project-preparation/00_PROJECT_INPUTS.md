## تجميع المدخلات الأولية للمشروع

### 1. معلومات المشروع العامة

| البند | القيمة |
|---|---|
| اسم المشروع | نظام إدارة الشيكات (Checks Management System) |
| نوع المشروع | تطبيق ويب بسيط — MVP |
| الفكرة | تطبيق لتسجيل ومتابعة الشيكات الصادرة والواردة |
| الجهة المالكة | غير محدد |
| المستخدمون المستهدفون | جهة إدارية تحتاج متابعة الشيكات |
| المشكلة | الحاجة إلى تتبع الشيكات الصادرة والواردة، حالاتها، وتواريخ استحقاقها بطريقة منظمة |
| المخرجات الرئيسية | تسجيل شيك، متابعة حالة، طباعة كشف بسيط |

| البند | القيمة |

### 4. الافتراضات الأولية

| الرقم | الافتراض |
|---|---|
| 1 | التطبيق سيُستخدم من قبل مستخدمين داخليين في جهة إدارية واحدة |
| 2 | لا يوجد فروع أو شركات متعددة في النسخة الأولى |
| 3 | الشيكات قد تكون صادرة (مدفوعة منا) أو واردة (مقبوضة لنا) |
| 4 | حالة الشيك تحتاج متابعة: مسجل، مستلم، مصرف، مرتجع، ملغي |
| 5 | الكشف المطلوب هو طباعة بسيطة وليس تقريرًا محاسبيًا معقدًا |
| 6 | لا يوجد طباعة شيكات فعلية — فقط تسجيل ومتابعة |

---

### 5. المعلومات الناقصة

| المعلومة الناقصة | الأثر | هل يمكن المتابعة مؤقتًا؟ | الافتراض المؤقت |
|---|---|---|---|
| لغة البرمجة والتقنيات | يؤثر على المعمارية وبدء التنفيذ | نعم — إلى مرحلة Technical Architecture | سيتم اقتراحها لاحقًا بقرار تيرا |
| نوع قاعدة البيانات | يؤثر على التصميم الفني | نعم — إلى مرحلة Data Design | سيتم اقتراحها لاحقًا |
| عدد المستخدمين المتوقع | يؤثر على الأداء | نعم — تطبيق صغير | أقل من 10 مستخدمين (افتراض) |
| هل يوجد صلاحيات أدوار | يؤثر على تصميم الشاشات | نعم — يمكن افتراض صلاحيات بسيطة | مدير + مستخدم |
| الجهة المستخدمة (شركة/مؤسسة) | غير مؤثر على التحليل حاليًا | نعم | جهة إدارية |

---

### 6. أسئلة تحتاج توضيحًا لاحقًا

1. هل يحتاج النظام إلى صلاحيات متعددة (مشرف، مدخل، مدقق) أم يكفي صلاحية واحدة؟
2. هل ستتم طباعة الشيكات فعليًا أم فقط طباعة كشف إدارة؟
3. هل يوجد بنوك متعددة لكل شركة أم بنك واحد فقط؟
4. هل يوجد عملاء/موردون مرتبطون بالشيكات أم فقط جهة نصية؟
5. هل المطلوب إشعارات قبل تاريخ الاستحقاق؟
